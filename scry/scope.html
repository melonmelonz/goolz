00 Status & roadmap. Updated 2026-05-11

Two engines, one shell — but they are not the same project. The live site at goolz.org/scry is a single app with a top-bar mode picker. V1 · PURE is the graded class deliverable. V2 · RUST · WASM is a side project that explores the same problem in a different stack.

V1 · PURE · the class demo. Hand-rolled vanilla JS, HTML, CSS. No framework. No bundler. No runtime dependency. Ships all five modules (Inspect, Hex, Disasm, Emu, Trace) on the synthetic-bus spine in §02. ELF parser is bounds-checked, drag-drop is size-capped, sample-manifest entries are whitelisted. This is the surface that must work for the demo.

V2 · RUST · WASM · the side project. Rust core compiled to WebAssembly, Svelte 5 + Vite shell. Currently ships Inspect and Hex against the Rust core via goblin. Not on the critical path; included because the two-stack contrast is the most honest way to show what the framework-free approach buys you. If V2 isn’t finished by demo day, V1 is still the deliverable.

Shared chrome. Dark mode, mode persistence, the file-context rail, and the workbench shell live at the wrapper level. Flipping engines feels like flipping a switch on the same instrument — intentional, because the point of having both is to compare them.

Next on V2. Port the V1 RV32IMA decoder and interpreter into Rust so V2’s Disasm and Emu panes light up. Then re-route the synthetic bus stream through the Rust core so V2 has its own Trace pane. Stretch: real .sal capture ingest, headers-only PE/Mach-O inspect.

Direction shifts from the original spec. Two builds instead of one (V2 added as the side project), dark mode now in scope, x86_64 disassembly dropped, synthetic-bus tracing replacing Saleae ingest for the demo. Everything else in this document still holds.

01 What it is. Premise

Scry is a browser-based systems inspection workbench. You drop a file. It parses it, disassembles it, emulates it, and shows you the bus traffic it would have generated on real hardware. Nothing is uploaded. Nothing is logged. The tab does the work.

It pairs with PEEK in temperament — quiet, observational, restrained — but moves into daylight. Where PEEK is a small creature in the dark, Scry is the bench you watch it on.

The point is depth, not breadth. One coherent demo, four modules that earn their place, a clean architecture that admits future modules without a rewrite.

02 The spine. Demo Flow

One end-to-end flow that the entire week is organized around. If this works, v1 ships. If a module slips, we cut depth, never the spine.

Userdrops peek-firmware.elf onto the page

Userclicks `main` → Disasm opens at that address

04 The modules. Five Views

Each module owns one view, one job. They communicate through the shell's file context and a small event channel for cross-module signals (Disasm asking Emu to jump to a PC, Emu pushing bus events to Trace).

04.3 Disasm — instructions resolved. Rust · iced-x86 + RV32

Linear listing with addresses, raw bytes, mnemonic, operands. Jump arrows in the gutter for short forward and backward branches. Symbols resolve inline — a call to spi_write_byte shows the name, not the address. RV32 decoding shares tables with the emulator so the two views always agree on what an instruction does.

08 Who it’s for. User persona

Scry is built for one kind of person and one kind of moment: a systems-leaning engineer who has a binary in front of them, no time, no toolchain on this laptop, and a need to know what is this and what would it do before deciding whether to spin up the heavy tools.

Primary The triaging engineer Reverse-engineer · firmware dev · security researcher
Context Has an unknown ELF, a captured bus log, or someone else’s firmware image. Wants the first 80% of triage answered in a browser tab, on any machine, without an install.
Goals Confirm format and architecture. See sections, symbols, entry. Pivot from a function name into a readable disassembly. Run it just enough to watch what it touches. Decode the bus traffic it would have produced.
Frustrations Web-based binary tools that upload files. Native tools that need a toolchain per architecture. Inspectors that show the header and stop. Disassemblers without an emulator. Emulators that don’t show MMIO.
Comfort Reads hex. Knows what an MMIO store looks like. Can spot a SPI mode 0 transaction in a waveform. Talks at the level of segments, syscalls, jump tables, decoder state machines.
Won’t accept Uploads. Sign-ups. Telemetry. UI that hides the actual bytes. “Magic” features that don’t show their work. A tool that can’t be opened from file://.
Secondary The curious systems student CS undergrad · CTF player · hardware-curious
Context Wants to feel how ELF, RV32, and a synthetic bus actually fit together. Came in through a class, a CTF, or a friend’s rabbit hole.
Goals Drop a sample. Click around. Walk from header bytes through disassembly into an emulator into a bus trace. Build intuition without spending three weekends on a toolchain.
Comfort Knows what hex is. Doesn’t know what e_phoff means yet. Will read a hover tip. Will not read a wiki.
Won’t accept Tools that require a paid plan, a Discord, or a 40-page setup guide to see a hello-world disassembly.

09 User stories. Shipped · acceptance-linked

Every story below is satisfied by code that is live at goolz.org/scry today. Each maps to one or more modules and to the security and UX passes documented in §00 and §13.

US-01 · Inspect As a reverse engineer, I want to drop an ELF on the page and immediately see its class, machine, entry point, sections, and symbols, so that I can decide in seconds whether this binary is worth deeper analysis. MeetsV1 Inspect · V2 Inspect (goblin) · file-context rail
US-02 · Privacy As a security-conscious user, I want every byte to stay on my machine, so that I can analyze sensitive customer or research artifacts without an upload, a login, or a server round-trip. MeetsClient-only architecture · status bar “LOCAL · NO UPLOAD” · CSP-friendly
US-03 · Robustness As a triaging engineer, I want a malformed or hostile ELF to fail loudly instead of crashing the tab, so that I can keep working without losing context on the other files I’ve already opened. MeetsBounds-checked DataView reader · section/segment/symbol caps · 64 MiB ingest cap
US-04 · Hex view As a forensic reader, I want to scroll a 10 MB+ binary in the hex view without jank and click any byte to learn what struct field it’s part of, so that I can correlate raw bytes with parsed structure. MeetsV1 virtualized hex · ELF header overlay · click-to-select byte detail
US-05 · Disasm pivot As a reverse engineer, I want to click a symbol in the Inspect view and land on that exact instruction in Disasm, so that I can read code at the level of intent (“main”) instead of addresses. MeetsV1 Inspect→Disasm pivot · goto-address form · symbol inlining
US-06 · Emulation As an embedded firmware engineer, I want to step or run an RV32IMA program in the tab and watch the register file and MMIO region change, so that I can verify behavior without booting real hardware. MeetsV1 RV32IMA interpreter · register + memory panels · STEP / RUN / PAUSE / RESET
US-07 · Bus decode As a hardware hacker, I want the emulator’s MMIO writes to surface as a decoded SPI/I²C waveform with click-for-detail on each transaction, so that I can read the bus the firmware would have produced. MeetsV1 Trace · SPI + I²C decoders · Canvas waveform · transaction detail panel
US-08 · Two engines As an engineering reviewer, I want to flip between a hand-rolled zero-dependency build and a Rust→WASM build of the same workbench, so that I can judge both the primitives and the production shape from one URL. MeetsUnified shell · hash routing (#v1 / #v2) · scry-mode persistence
US-09 · Dark mode As a long-session reader, I want a dark theme that persists across page loads and across both engines, so that I can read hex and disassembly for an hour without eye strain. MeetsCSS tokens · scry-theme in localStorage · postMessage across iframe · prefers-color-scheme fallback
US-10 · Trusted samples As a first-time visitor, I want one click to load a sample firmware and see the spine end-to-end, so that I can evaluate the tool without owning a binary. MeetsSamples picker · filename whitelist regex · bundled peek-firmware.elf
US-11 · Receipts As an applications reviewer, I want a single document that says what was built, why, what the AI did, and what the human did, so that I can evaluate the work without spelunking the git history. MeetsThis scope dock · §00 status · §13 build-with-Claude writeup · §12 verbatim transcript
US-12 · Local first As a distrust-by-default user, I want the V1 build to work from file:// with no network, so that I can verify the tool is doing what it claims on a fully airgapped machine. MeetsV1 zero runtime deps · no CDN · offline-capable assets

13 Built with Claude. AI collaboration writeup

This section is for the class about AI. It is the same content shipped as docs/ai-session-writeup.md in the repo, surfaced here so the public scope dock is the one document anyone needs to read to understand both what scry is and how it was built.

The tool: Claude Code (Opus 4) in a terminal on Bazzite Fedora. The collaborator: me. The artifact you are reading: a record of where the model carried weight, where I steered, and where the seams are.

13.1 What the AI did.

Across one continuous Claude Code session the assistant: diagnosed why all tabs except EMPTY appeared greyed out (intentionally disabled until a file loads; verified from the live site, not by guessing); added a tokenized dark mode across the landing, V1 workbench, scope doc, and V2 shell, with a synchronous head script so the first paint matches the user's stored choice; did a security pass that bounds-checked every DataView access in the ELF parser, capped section/segment/symbol counts to defend against integer-overflow DoS, whitelisted sample-manifest filenames by regex, and capped drag-drop ingestion at 64 MiB; did a UX pass that put tooltips on disabled tabs, hints in the status bar per module, color-coded state badges on the emulator, click-to-select hex byte detail, a goto-address form in disasm, click-to-pivot from inspect symbol to disasm address, and click-for-detail on trace transactions; scaffolded the entire V2 stack from zero (Cargo workspace, scry-core crate, wasm-pack output wired through Vite + Svelte 5); chained Rust + Vite + esbuild in scripts/build.sh; and replaced the landing chooser with a unified shell that hot-swaps V1 and V2 inside an iframe and pushes theme changes via postMessage.

Along the way it hit one real bug: wasm-opt refused to validate the modern bulk-memory ops Rust emits. The model diagnosed it from the actual error text and fixed it by disabling the post-opt step in Cargo.toml. The build broke once. The fix took two minutes. That is the failure mode I want from an AI tool.

13.2 How I worked with it.

I treated Claude like a junior engineer who reads fast, writes faster, and has zero context on my taste until I correct it. My prompts were short and directional: "do v2 now please", "fix all of those please", "UI and UX need considered". When the model defaulted to verbose explanations or heavy frameworks, I cut it off. When it suggested something I would not ship (a CDN script tag, an em-dash in prose, a placeholder TODO inside a deployed file), I said so once and the rule stuck for the rest of the session.

The thing I did not delegate was scope. Every time the assistant proposed "more features", I either took the suggestion as-is or replaced it with something I actually wanted. The model is good at filling in the next reasonable step. It is not good at deciding which step the project needs. That is on me.

13.3 Where AI was load-bearing.

Plumbing. Cargo workspaces, Vite config for WASM, Svelte 5 runes syntax, hand-writing the bounds-checked binary reader. This is the boring code where you can lose half a day to a typo. Claude got the shape right on the first pass for almost all of it.

Token hygiene. Dark mode meant touching every CSS file. Claude caught more #hex literals I had missed than I would have on my own, and it kept the names consistent across the three surfaces.

The audit pass. I asked for "safety and security". The model came back with three findings ranked by exploitability, with code for each fix. I would not have thought to whitelist sample filenames on my own.

13.4 Where I had to steer.

Aesthetic restraint. The model wanted to add icons, framed panels, animations. I kept saying "subtlety". The output is now subtle. This took several rounds.

Avoiding "more is better". When I said "flesh it all out", I had to be specific about what flesh out meant or the model would have added a fifth tab and a settings panel nobody asked for.

Voice. The model defaults to corporate documentation prose. My voice is hedged and concrete. I wrote the prose. Claude wrote the scaffolding underneath.

13.5 Friction worth naming.

The model occasionally needed to be told to use its dedicated file-reading tools instead of shell cat | grep. The system prompt enforces this; the assistant sometimes ignores it. Minor.

The model has no view of the live deployed site. Twice in this session I had to tell it "the site does X" so it would stop assuming local source matched production. The fix was always to curl the live URL and compare. Worth remembering.

The V2 build broke the first deploy because Vite was configured with base: '/v2/' but on goolz.org V2 lives at /scry/v2/. Asset URLs resolved to the wrong path. Fix: switch to base: './' so the build is location-independent. Two minutes once we knew.

13.6 What I would do differently.

I would have written the security pass first, not after the UX pass. The bounds-check changes touched code the UX pass had already moved.

I would have set up the Vite + Svelte V2 scaffold on day one. V1 was fun, but V1 is also a re-implementation of goblin, and the only reason to do that is the receipt that I can.

I would have asked for a deliverable like this one earlier. Writing it in the same session as the build means I remember why each decision was made.

13.7 Closing.

The work product is at goolz.org/scry. The source is at github.com/melonmelonz/scry. If you read the V1 commits in order, you can see the build, including the moments where I changed direction. None of the code in this repo is generated and pasted blind. Every file got read, edited, and committed by a human who can defend the choices in it.

The AI is part of the toolchain, sitting at roughly the same level as my editor and my compiler. It would not have written scry without me. I would have shipped a worse, slower, less-tested scry without it.
