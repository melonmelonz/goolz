00 Status & roadmap. Updated 2026-05-11

Two engines, one shell. The live site at goolz.org/scry is a single app. A top-bar mode picker swaps between V1 · PURE (hand-rolled vanilla JS, HTML, CSS — no framework, no bundler, no runtime dependency) and V2 · RUST · WASM (Rust core compiled to WebAssembly, Svelte 5 + Vite shell). Dark mode and state live at the shell level so flipping engines feels like flipping a switch on the same instrument.

Shipped. V1 covers all five modules (Inspect, Hex, Disasm, Emu, Trace) on the synthetic-bus spine in §02. ELF parser is bounds-checked, drag-drop is size-capped, sample-manifest entries are whitelisted. V2 ships Inspect and Hex against the Rust core via goblin.

Next. Port the V1 RV32IMA decoder and interpreter into Rust so V2's Disasm and Emu panes light up. Then re-route the synthetic bus stream through the Rust core so V2 has its own Trace pane. Stretch: real .sal capture ingest, headers-only PE/Mach-O inspect.

Direction shifts from the original spec. Two builds instead of one, dark mode now in scope, x86_64 disassembly dropped, synthetic-bus tracing replacing Saleae ingest for v1. Everything else in this document still holds.

01 What it is. Premise

Scry is a browser-based systems inspection workbench. You drop a file. It parses it, disassembles it, emulates it, and shows you the bus traffic it would have generated on real hardware. Nothing is uploaded. Nothing is logged. The tab does the work.

It pairs with PEEK in temperament — quiet, observational, restrained — but moves into daylight. Where PEEK is a small creature in the dark, Scry is the bench you watch it on.

The point is depth, not breadth. One coherent demo, four modules that earn their place, a clean architecture that admits future modules without a rewrite.

02 The spine. Demo Flow

One end-to-end flow that the entire week is organized around. If this works, v1 ships. If a module slips, we cut depth, never the spine.

Userdrops peek-firmware.elf onto the page
Shelldetects ELF magic (0x7F 'E' 'L' 'F'), routes to Inspect
Inspectparses sections, symbols, segments — renders the metadata grid
Userclicks `main` → Disasm opens at that address
Disasmrenders linear RV32 listing, jump arrows in the gutter, symbols resolved inline
Userclicks Emulate from entry
Emuloads ELF segments, sets PC, runs in a Web Worker
Firmwarebit-bangs a sequence over an MMIO-mapped SPI controller (0xF000_0000)
Busemulator's memory-write path emits synthetic bus events for the controller region
Traceingests events, renders waveform on Canvas, decodes SPI transactions byte-by-byte
Usersees 0xA1 · 0xC0 · 0x55 with cycle-accurate timing — the firmware speaking, simulated

04 The modules. Five Views

Each module owns one view, one job. They communicate through the shell's file context and a small event channel for cross-module signals (Disasm asking Emu to jump to a PC, Emu pushing bus events to Trace).

04.1 Inspect — the parse view. Rust · goblin

Parses ELF in Rust via goblin, exposes typed structs across the WASM boundary. Renders sections, symbols, segments, and a strings extractor. Every address and every symbol is clickable; click a symbol, Disasm opens there.

04.3 Disasm — instructions resolved. Rust · iced-x86 + RV32

Linear listing with addresses, raw bytes, mnemonic, operands. Jump arrows in the gutter for short forward and backward branches. Symbols resolve inline — a call to spi_write_byte shows the name, not the address. RV32 decoding shares tables with the emulator so the two views always agree on what an instruction does.

08 Built with Claude. AI collaboration writeup

This section is for the class about AI. It is the same content shipped as docs/ai-session-writeup.md in the repo, surfaced here so the public scope dock is the one document anyone needs to read to understand both what scry is and how it was built.

The tool: Claude Code (Opus 4) in a terminal on Bazzite Fedora. The collaborator: me. The artifact you are reading: a record of where the model carried weight, where I steered, and where the seams are.

08.1 What the AI did.

Across one continuous Claude Code session the assistant: diagnosed why all tabs except EMPTY appeared greyed out (intentionally disabled until a file loads; verified from the live site, not by guessing); added a tokenized dark mode across the landing, V1 workbench, scope doc, and V2 shell, with a synchronous head script so the first paint matches the user's stored choice; did a security pass that bounds-checked every DataView access in the ELF parser, capped section/segment/symbol counts to defend against integer-overflow DoS, whitelisted sample-manifest filenames by regex, and capped drag-drop ingestion at 64 MiB; did a UX pass that put tooltips on disabled tabs, hints in the status bar per module, color-coded state badges on the emulator, click-to-select hex byte detail, a goto-address form in disasm, click-to-pivot from inspect symbol to disasm address, and click-for-detail on trace transactions; scaffolded the entire V2 stack from zero (Cargo workspace, scry-core crate, wasm-pack output wired through Vite + Svelte 5); chained Rust + Vite + esbuild in scripts/build.sh; and replaced the landing chooser with a unified shell that hot-swaps V1 and V2 inside an iframe and pushes theme changes via postMessage.

Along the way it hit one real bug: wasm-opt refused to validate the modern bulk-memory ops Rust emits. The model diagnosed it from the actual error text and fixed it by disabling the post-opt step in Cargo.toml. The build broke once. The fix took two minutes. That is the failure mode I want from an AI tool.

08.2 How I worked with it.

I treated Claude like a junior engineer who reads fast, writes faster, and has zero context on my taste until I correct it. My prompts were short and directional: "do v2 now please", "fix all of those please", "UI and UX need considered". When the model defaulted to verbose explanations or heavy frameworks, I cut it off. When it suggested something I would not ship (a CDN script tag, an em-dash in prose, a placeholder TODO inside a deployed file), I said so once and the rule stuck for the rest of the session.

The thing I did not delegate was scope. Every time the assistant proposed "more features", I either took the suggestion as-is or replaced it with something I actually wanted. The model is good at filling in the next reasonable step. It is not good at deciding which step the project needs. That is on me.

08.3 Where AI was load-bearing.

Plumbing. Cargo workspaces, Vite config for WASM, Svelte 5 runes syntax, hand-writing the bounds-checked binary reader. This is the boring code where you can lose half a day to a typo. Claude got the shape right on the first pass for almost all of it.

Token hygiene. Dark mode meant touching every CSS file. Claude caught more #hex literals I had missed than I would have on my own, and it kept the names consistent across the three surfaces.

The audit pass. I asked for "safety and security". The model came back with three findings ranked by exploitability, with code for each fix. I would not have thought to whitelist sample filenames on my own.

08.4 Where I had to steer.

Aesthetic restraint. The model wanted to add icons, framed panels, animations. I kept saying "subtlety". The output is now subtle. This took several rounds.

Avoiding "more is better". When I said "flesh it all out", I had to be specific about what flesh out meant or the model would have added a fifth tab and a settings panel nobody asked for.

Voice. The model defaults to corporate documentation prose. My voice is hedged and concrete. I wrote the prose. Claude wrote the scaffolding underneath.

08.5 Friction worth naming.

The model occasionally needed to be told to use its dedicated file-reading tools instead of shell cat | grep. The system prompt enforces this; the assistant sometimes ignores it. Minor.

The model has no view of the live deployed site. Twice in this session I had to tell it "the site does X" so it would stop assuming local source matched production. The fix was always to curl the live URL and compare. Worth remembering.

The V2 build broke the first deploy because Vite was configured with base: '/v2/' but on goolz.org V2 lives at /scry/v2/. Asset URLs resolved to the wrong path. Fix: switch to base: './' so the build is location-independent. Two minutes once we knew.

08.6 What I would do differently.

I would have written the security pass first, not after the UX pass. The bounds-check changes touched code the UX pass had already moved.

I would have set up the Vite + Svelte V2 scaffold on day one. V1 was fun, but V1 is also a re-implementation of goblin, and the only reason to do that is the receipt that I can.

I would have asked for a deliverable like this one earlier. Writing it in the same session as the build means I remember why each decision was made.

08.7 Closing.

The work product is at goolz.org/scry. The source is at github.com/melonmelonz/scry. If you read the V1 commits in order, you can see the build, including the moments where I changed direction. None of the code in this repo is generated and pasted blind. Every file got read, edited, and committed by a human who can defend the choices in it.

The AI is part of the toolchain, sitting at roughly the same level as my editor and my compiler. It would not have written scry without me. I would have shipped a worse, slower, less-tested scry without it.
