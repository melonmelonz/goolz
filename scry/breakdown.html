V1 · V2 · THE SAME UI, TWO ENGINES

Two implementations of the same workbench: pure JavaScript on the left, Rust compiled to WebAssembly on the right.

Same panes, same shortcuts, same files. The point of having two engines isn't to ship both forever — it's to make the cost and shape of each approach legible. This page lines them up.

01 Architecture at a glance SHELL · ENGINES 
  scry.goolz.org/
  ┌── index.html      ← unified shell. iframes one engine at a time.
  │     V1 • V2 toggle · theme · tech breakdown
  │
  ├── v1/              pure JS · no bundler · ~50 source files
  │     hex · inspect · disasm · emu · trace · wave · game
  │
  └── v2/              Rust → wasm-pack → Svelte 5
        scry-core.wasm (one file) + Svelte components
The shell is the contract. Both engines accept the same drag-and-drop file, get the same theme toggle pushed in via postMessage, and live behind the same URL. The toggle at the top of the page swaps which one is mounted — nothing else.
02 What each engine actually is SIDE BY SIDE
V1 · PURE
Vanilla JS, nothing to install.

ES modules, classic <script type="module">. No bundler, no build step, no framework. Files are served byte-for-byte the way they sit in web/v1/.

  • State: a hand-rolled pub-sub store (fileStore, router, etc).
  • DOM: a 30-line el(tag, props, children) helper. Never innerHTML.
  • ELF parser: hand-written, ~400 LOC, just the fields the UI uses.
  • WAV parser: hand-written RIFF chunk walker.
  • GBA emulator: vendored endrift/gbajs (BSD-2). It runs the cart for real.
  • Disasm/EMU: a small RV32IMA decoder + interpreter, written by hand.

Source size minified: a few hundred KB. Cold-start: one HTTP round-trip per module the browser actually asks for. No WASM, no SharedArrayBuffer, no service worker.

V2 · RUST·WASM
Rust core, Svelte 5 shell.

Crate scry-core compiles to a single .wasm via wasm-pack. Svelte 5 (runes mode) calls into it for parsing and decoding.

  • State: Svelte 5 runes ($state, $derived, $effect).
  • ELF parser: goblin crate — full-fat, battle-tested.
  • WAV decoder: Rust RIFF walker, produces a real Float32 sample buffer the browser plays via Web Audio.
  • GBA cart: Rust verifies the Nintendo header checksum (sum bytes 0xA0..=0xBC, negate, subtract 0x19).
  • Emulator: not yet ported — V1 owns that story for now.

Source size: the wasm module is the bulk — one fetch, then everything runs in-process. The Vite/Svelte build bundles + minifies + tree-shakes.

03 Pane-by-pane status WHAT WORKS, WHERE
INSPECT (ELF)
V1: hand-rolled parser, all critical fields.
V2: goblin — richer, surfaces more sections/symbols.
HEX
V1: virtualized; 64 MiB cap; click-to-overlay.
V2: same UX, entropy strip driven by scry-core::entropy_blocks.
WAVE (audio)
V1: hand-rolled PCM walk → Float32 + Web Audio + 256-bucket envelope.
V2: Rust decode_wav returns the buffer; UI is symmetric.
DISASM (RV32)
V1: hand-rolled RV32IMA decoder, register/mnemonic view.
V2: punt for now — V1 covers this; Rust port is on the roadmap.
EMU (RV32)
V1: step/run with MMIO peripherals (SPI, I2C).
V2: not yet.
GAME (GBA)
V1: vendored gbajs renders to a canvas — cart actually runs.
V2: CART pane — Rust verifies the header checksum.
04 Trade-offs, honestly WHY BOTH
WHEN V1 WINS
  • Zero compile chain. Edit a JS file, refresh, done.
  • Smaller initial payload for small files — no wasm fetch.
  • Easier to vendor JS-native libraries (the emulator is the perfect example).
  • Easier to read end-to-end as a teaching artifact.
WHEN V2 WINS
  • Real parsers. goblin handles weird ELFs V1 won't.
  • Type-safe core. Rust + serde keeps the JS layer thin.
  • Native-ish speed for big files — envelope/entropy in wasm beats JS by 5-20x.
  • Shareable core. The same crate could back a CLI or a desktop app.
The unifying constraint: nothing leaves the browser tab. Both engines parse, render, and emulate locally. No upload, no telemetry, no service worker, no login. The user's binary stays on their machine.
