Edge Cases

Exercise 5 / 5 . Level 3 . 30 min

The schema works when everything goes right. Now: what happens when it doesn't?

Scenario 1: User deletes their account

User #7 decides they're done. They hit "Delete Account." What happens to their data?

Tables affected

progress -- every row where user_id = 7
questions -- every row where user_id = 7
resources -- every row where created_by = 7 (if they were a case worker)
assignments -- every row where caseworker_id = 7 or client_id = 7

My decision: Soft delete

Add an is_active column to the users table. Default true. When they "delete" their account, flip it to false. Don't actually remove anything.

Why: cascading delete is too destructive. If a case worker deletes their account, wiping all the resources they added hurts every other user who relies on those links. If a returning citizen deletes their account, their progress data might still be useful for aggregate reporting ("how many people completed module 3 this quarter?"). Soft delete keeps the data while hiding the person.

For the AI chat history (questions), I'd actually let the user hard-delete those if they want. That's personal. Someone who was incarcerated has a real reason to want their conversation history gone. So: soft delete the user row, offer hard delete of their questions.

Scenario 2: Duplicate data

Two users sign up with the same email. Should the database allow it?

No. UNIQUE constraint on users.email.

Already handled this in Exercise 3 (the AI caught it). Without it, you could have two accounts for jane@email.com and the login system wouldn't know which one to authenticate. Password reset would break. The database should reject the second signup at the database level, not just in the app code. Defense in depth -- even if the app code has a bug, the database itself won't allow it.

What about duplicate glossary terms?

Also add UNIQUE on glossary_terms.term. Two different definitions for "cloud" would confuse the AI and the user. If someone wants to update a definition, that's an UPDATE, not a second INSERT.

What about duplicate progress entries?

Tricky. Should a user be able to have two progress rows for the same step? No. Add a UNIQUE constraint on the combination of (user_id, step_id). One user, one progress row per step. If they complete it, that's an UPDATE on the existing row, not a new row. This also prevents the app from accidentally double-counting completions.

Scenario 3: Required vs. optional fields

Table Column Required? Why
users name Required Case workers need to see who they're working with. Can't have a nameless row.
users email Required Used for login. No email, no account.
users role Required Determines what they can access. Default to 'citizen' if not specified.
modules title Required A module without a title is useless in a list.
modules description Optional Nice to have, but a module can function without it.
modules difficulty Optional Default to 'beginner' if not set. Not a showstopper.
resources url Required A resource without a link is just a sentence. The link is the point.
resources category Required Without it, the resource can't be filtered. Default to 'general' if needed.
resources description Optional Some resources are self-explanatory from the title.
progress completed_at Optional Null until the step is completed. That's the whole point of nullable here -- null means "hasn't happened yet."
questions ai_response Optional Could be null briefly while the AI is generating. The question exists before the answer does.
questions session_id Optional Standalone questions don't need a session. Only follow-ups do.
What I learned

Edge cases are design decisions, not bugs. There's no single right answer for what happens when a user gets deleted or whether a field is required. But there IS an unconsidered answer, and that's worse than any of the options. The soft delete decision for ReEntry Tech is specific to this app's context -- people coming out of incarceration have real privacy concerns, and case workers adding resources are contributing to a shared pool. Both of those facts shaped the design. A different app would make different choices.
