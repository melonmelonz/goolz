Yesterday was APIs -- front-end talks to back-end, JSON goes back and forth. Today: where does the data actually live? Every response the API sends has to come from somewhere. That somewhere is a database. Tables, rows, columns, keys. It's a spreadsheet that enforces rules and doesn't break at scale.

Morning Warm-up -- Where Does the Data Live?

App I picked: Spotify.

Data about ME: name, email, password (hashed hopefully), subscription tier, listening history, playlists I created, saved songs, followed artists, what I'm playing right now, device info, play counts, skip counts probably. There's definitely a table somewhere that knows I listened to the same Nujabes album forty times last month.

Data that ISN'T about me: songs (title, artist, album, duration, audio file URL, genre tags), artists (name, bio, monthly listeners, discography), albums (name, release date, track list, cover art), playlists made by other people, podcast episodes.

Delete and reinstall: everything is still there. My playlists, my history, my queue. Because none of that lives on my phone. It lives in Spotify's database. My phone is just the front-end making API calls. The data survived because it's persistent -- stored server-side, not in local memory.

Verbal Mini-Demo

Pointed to the user_id column in my progress table. It's a foreign key that connects to users.id. Without it, the progress table is just a pile of "step 4 is complete" rows with no way to know WHO completed it. Every row in progress would be anonymous. The case worker dashboard user story -- "I want to see which clients are stuck" -- requires knowing which user each progress row belongs to. That foreign key is the entire link.

Partner asked: "Why not just put the progress data directly in the users table?" Because one user has many progress entries -- one per step per module. If I crammed that into the users table, I'd need columns like module_1_step_1_complete, module_1_step_2_complete, and that's insane. It doesn't scale. A separate progress table with a foreign key lets me have as many rows as needed without changing the schema.
