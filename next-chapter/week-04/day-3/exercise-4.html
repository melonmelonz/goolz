Peer Review Round 2

Exercise 4 / 5 . Level 2 . 20 min

Different partner this time. They got the updated schema (post-AI review). Tougher questions -- not just "does the data exist" but "can you actually do things with it."

Questions they asked me

"If I wanted to search for all resources about transportation, which table and column would I look in?"

The resources table, filtering by the category column where category = 'transportation'. Before the AI review I wouldn't have been able to answer this because I didn't have a category column. I would've had to search the description text for the word "transportation" which is slow and unreliable.

"If I wanted to show a list of modules sorted by difficulty, can your schema do that?"

Yes. The modules table has a difficulty column. Sort by it. But there's a wrinkle: difficulty is stored as text ('beginner', 'intermediate'), and alphabetical sort would put 'beginner' before 'intermediate' which happens to be correct. But if I add 'advanced' later, alphabetical gives me 'advanced', 'beginner', 'intermediate' -- wrong order. I'd either need to store difficulty as a number (1, 2, 3) or define a custom sort order. For now with two levels it works, but it's a landmine.

"If a user completes step 3 of module 1, what rows get created or changed?"

One row gets created in the progress table: user_id = that user, step_id = the ID of step 3 in module 1, completed = true, completed_at = now. If a progress row for that user/step combo already exists with completed = false, then it gets updated instead of created -- flip completed to true and set completed_at. No other tables change. The modules table, the steps table, the users table -- all untouched. Progress is the only table that moves.

"How would the case worker dashboard work? Walk me through the query."

Start in assignments: get all rows where caseworker_id = the logged-in case worker. That gives me a list of client IDs and module IDs. For each client+module pair, look up the module's steps (from the steps table where module_id matches), then check the progress table for that client's entries on those steps. Steps with no progress row or completed = false = that's where they're stuck. Three tables touched: assignments, steps, progress. Plus users to get the client's name for display.

Questions I asked my partner

Their app

They're building a community events board -- people post events, others RSVP. I asked: "If an event gets cancelled, what happens to the RSVPs?" They hadn't thought about it. Their rsvps table has event_id as a foreign key, but no status column on the events table. They'd need an is_cancelled column or a status column ('active', 'cancelled') so the app knows not to show it. The RSVPs would need to be handled too -- notify people, or just hide the event and let the RSVPs become orphans. Same kind of edge case thinking we're doing in Exercise 5.
