AI-Assisted Schema Review

Exercise 3 / 5 . Level 2 . 30 min

Pasted my schema into ChatGPT with the prompt template from the assignment. Described ReEntry Tech as "an AI-powered digital literacy tool for people returning from incarceration, with guided modules, progress tracking, AI Q&A, a glossary, and local community resources."

What AI suggested

Suggestion 1: Add an updated_at column to every table

AI said

"You have created_at on every table but no updated_at. If a case worker edits a resource's URL, you'll want to know when that change happened. Add updated_at to at least resources, glossary_terms, users, and modules."

Decision: Accept

Fair point. If a resource URL changes, knowing when it was last updated matters. Case workers need to know if a resource link is stale. Adding updated_at to resources, glossary_terms, users, and modules. Not adding it to progress or questions -- those are events, not things you edit.

Suggestion 2: Add a category column to resources

AI said

"Your resources table has title, URL, and description, but no category. If a returning citizen is looking for transportation help versus job listings versus benefits, they'd need to filter. Add a category column (e.g., 'transportation', 'employment', 'benefits', 'housing')."

Decision: Accept

This one is obvious in hindsight. My Day 2 API design already had GET /resources returning all resources, but with no category column there's no way to filter them. A returning citizen looking for bus info shouldn't have to scroll through thirty links about housing assistance. Adding category as a text column.

Suggestion 3: Create a sessions table for AI conversations

AI said

"Your questions table stores individual Q&A pairs, but there's no concept of a conversation. If a user asks a follow-up question, the AI has no context. Create a sessions table and link each question to a session, so the AI can maintain conversational context."

Decision: Modify

The idea is right but a whole sessions table feels like over-engineering for a Phase 2 prototype. Instead, I'll add a session_id column to the questions table. It groups related questions together without a separate table. Multiple questions with the same session_id = one conversation. Simple. If I need session metadata later (start time, which module they were in), I can pull it out into its own table then.

Suggestion 4: Add a unique constraint on users.email

AI said

"Nothing in your schema prevents two users from registering with the same email. Add a unique constraint on the email column in the users table."

Decision: Accept

Didn't even think about this. Two people with the same email would break login, password reset, everything. Email has to be unique. This is the kind of thing that's invisible until it's a catastrophe. Marking email as UNIQUE.

Suggestion 5: Add a "tags" system for modules using a join table

AI said

"Consider adding tags to modules (e.g., 'beginner', 'smartphone', 'email', 'employment') with a many-to-many relationship via a module_tags join table. This lets users browse modules by topic."

Decision: Reject

I already have a difficulty column on modules. For a Phase 2 prototype with maybe 10-15 modules, a full tagging system with a join table is premature. Case workers can describe what each module covers in the description field. If the app grows to 100+ modules, yeah, tags would matter. But right now it's complexity I don't need and wouldn't demonstrate in the Phase 2 gate.

Takeaway

Three accepted, one modified, one rejected. The AI was most useful for the things I didn't think about -- the unique constraint on email and the missing category column. The session table idea was directionally right but over-scoped for this stage. The tags suggestion was pure scope creep. Accept/reject/modify is a good framework. It forces you to actually evaluate instead of just copying.
