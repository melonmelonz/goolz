Design Your App's Tables

Exercise 2 / 5 . Level 1 . 30 min

Starting from the user stories I wrote on Day 1. Each one implies data that needs a home. The core three: AI Q&A chat, guided walkthrough modules, and progress tracking. Plus: glossary, local resources, and case worker features.

The schema

USERS
-----------
id (PK)
name
email
role                 -- 'citizen' or 'caseworker'
created_at

MODULES
-----------
id (PK)
title                -- "Setting Up Email", "Using a Smartphone"
description
difficulty           -- 'beginner', 'intermediate'
created_at

STEPS
-----------
id (PK)
module_id (FK -> modules.id)
title                -- "Open the Gmail app", "Tap Create Account"
content              -- the actual instruction text
step_order           -- 1, 2, 3... display order within the module
created_at

PROGRESS
-----------
id (PK)
user_id (FK -> users.id)
step_id (FK -> steps.id)
completed            -- true/false
completed_at         -- timestamp, null if not completed yet

QUESTIONS
-----------
id (PK)
user_id (FK -> users.id)
body                 -- what the user typed
ai_response          -- what the AI sent back
created_at

GLOSSARY_TERMS
-----------
id (PK)
term                 -- "cloud", "two-factor authentication"
definition           -- plain-language explanation
created_at

RESOURCES
-----------
id (PK)
title                -- "County Benefits Portal", "City Bus App"
url
description
created_by (FK -> users.id)   -- the case worker who added it
created_at

ASSIGNMENTS
-----------
id (PK)
caseworker_id (FK -> users.id)
client_id (FK -> users.id)
module_id (FK -> modules.id)
assigned_at

User story check

"As a returning citizen, I want to ask questions in plain language"

Covered by the questions table. Each row stores what they asked (body) and what the AI said back (ai_response). Linked to the user by user_id.

"As a returning citizen, I want step-by-step guides"

Covered by modules + steps. A module is the guide ("Setting Up Email"). Steps are the individual instructions within it, ordered by step_order. One module has many steps.

"As a returning citizen, I want to track what I've learned"

Covered by the progress table. One row per user per step. completed = true means they finished that step. To see overall progress for a module, query all steps for that module and check which ones have progress rows for that user.

"As a returning citizen, I want explanations of jargon"

Covered by glossary_terms. Each row is one term with a plain-language definition. The AI can also pull from this table to ground its responses -- "here's what the glossary says, now let me explain it further."

"As a case worker, I want to see which clients are stuck"

Covered by progress + assignments. The case worker looks up their assigned clients, then checks each client's progress rows. If a client has completed = false on the same step for a while, they're stuck.

"As a case worker, I want to assign modules to clients"

Covered by the assignments table. It's a three-way link: this case worker assigned this module to this client. Both caseworker_id and client_id point to users.id -- same table, different roles.

"As a case worker, I want to add local resources"

Covered by resources. The created_by foreign key tracks which case worker added it. Title, URL, description. Simple CRUD.

Relationships

One-to-many

users -> questions: one user asks many questions.
users -> progress: one user has many progress entries.
modules -> steps: one module has many steps.
steps -> progress: one step has many progress entries (from different users).
users -> resources: one case worker creates many resources.

Schema swap feedback

Partner noticed I don't have a way to track which module a question was asked from. If someone is in the "Setting Up Email" module and asks "what's a password manager?", the questions table just stores the text -- no link to the module they were in. Good catch. I could add a module_id (FK, nullable) to the questions table to capture context. Not critical, but useful for the case worker dashboard.

Also pointed out: the glossary_terms table is standalone. No foreign keys, no relationship to other tables. That's fine -- it's a lookup table. Not everything needs to connect to everything.
