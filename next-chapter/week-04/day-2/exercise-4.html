Error Handling

Exercise 4 / 5 . Level 2 . 20 min

What happens when things go wrong? Let's find out by deliberately asking for stuff that doesn't exist.

Breaking things on purpose

JSONPlaceholder -- missing post

https://jsonplaceholder.typicode.com/posts/99999

What do you see? Is it an error?

An empty object: {}. Just curly braces with nothing inside. That's... not really an error? JSONPlaceholder doesn't return a 404 for missing IDs. It just hands back empty data and a 200 OK. A real production API would return a 404 with an error message. This is why JSONPlaceholder is a practice API, not a model for how to build one.

GitHub -- nonexistent user

https://api.github.com/users/thisisnotarealgithubusername12345678

What does the "message" field say?

"Not Found". That's a real 404. GitHub returns:

{
  "message": "Not Found",
  "documentation_url": "https://docs.github.com/rest",
  "status": "404"
}

This is better. Clear error message, a link to the docs, and the status code right in the JSON body. You can see it in the browser and in code. If my app was making this request, the .catch() or a status check would handle it.

Status Code Reference Card

Code Name What it means
200 OK Everything worked. Here's the data you asked for. The happy path.
201 Created Your request created something new. Like when you POST a new resource and the server says "got it, it exists now." Order placed.
400 Bad Request You sent something the server can't understand. Missing a required field, wrong data format, malformed JSON. "I can't read your handwriting, try again."
401 Unauthorized You haven't proven who you are. No login, no API key, expired token. "Who are you? Show me your ID."
403 Forbidden You proved who you are, but you still can't do this. Like a regular user trying to access admin settings. "I know who you are, but you're not allowed in here."
404 Not Found The thing you asked for doesn't exist. Wrong URL, deleted resource, typo in the endpoint. "We don't have that on the menu."
500 Server Error The server broke. Not your fault -- something crashed on their end. "The kitchen is on fire. Come back later."
The distinction that matters

401 vs 403: 401 means "I don't know who you are" (not logged in). 403 means "I know who you are and you can't do this" (logged in but no permission). For ReEntry Tech this matters: a returning citizen logged in shouldn't be able to DELETE a resource -- that's a 403, not a 401. They're authenticated, just not authorized for that action.

400 vs 404: 400 means the request itself is broken (bad syntax, missing data). 404 means the request was fine but the thing doesn't exist. If I send GET /modules/999 and module 999 doesn't exist, that's a 404. If I send POST /modules with no body at all, that's a 400.
