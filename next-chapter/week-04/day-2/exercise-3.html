HTTP Methods for ReEntry Tech

Exercise 3 / 5 . Level 2 . 25 min

From yesterday's design thinking: ReEntry Tech is an AI-powered digital literacy tool for people coming out of incarceration. The three core features are AI Q&A chat, guided walkthrough modules, and progress tracking. Today I'm designing the API endpoints that would make those features work.

Resources

What my app deals with

Modules -- the guided walkthroughs (how to set up email, use a smartphone, apply for a job online).
Progress -- where each user is in each module.
Questions -- what users ask the AI chatbot.
Glossary -- jargon definitions (what is "the cloud," what is "two-factor auth").
Resources -- local community resources that case workers can add (city bus app link, county benefits portal, etc.).
Users -- both returning citizens and case workers.

Endpoints

Method URL What it does
GET /modules Returns all available learning modules
GET /modules/3 Returns a single module (module #3) with its steps
POST /modules/3/progress Mark a step as complete for the current user
GET /users/7/progress Get user #7's progress across all modules (case worker view)
POST /questions Submit a question to the AI chatbot, get an answer back
GET /glossary Returns all glossary terms and definitions
GET /glossary?term=cloud Search the glossary for a specific term
GET /resources Returns all local community resources
POST /resources Add a new local resource (case workers only)
PUT /resources/12 Update resource #12 (fix a broken link, change hours)
DELETE /resources/12 Remove a resource that's no longer available
PUT /users/7/profile Update user #7's profile info

Thinking through it

What data does the user need to see? (GET)

Modules, their own progress, glossary definitions, and local resources. The case worker also needs to see other users' progress. That's a lot of GETs -- most of what the app does is showing information.

What data does the user need to create? (POST)

Questions to the AI chatbot (every time they ask something, that's a new request). Progress updates when they finish a step. Case workers creating new resource entries.

What data might need to be updated? (PUT)

Resource listings (URLs change, hours change). User profile info. Maybe module content itself if we want case workers to customize guides for their community.

What data might need to be deleted? (DELETE)

Resources that no longer exist (a program shuts down, a link goes dead). Maybe old questions from the AI chat history if the user wants to clear it.

Observation

The endpoints are just the same patterns repeated for different resources. Once you know GET/POST/PUT/DELETE and the URL pattern, you can design endpoints for basically anything. The hard part isn't the API design -- it's figuring out what the resources are in the first place. That's why yesterday's design thinking mattered.
