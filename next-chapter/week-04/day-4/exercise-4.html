AI-Generated SQL Review

Exercise 4 / 5 . Level 2 . 30 min

Same drill as Day 3's schema review. Paste the schema into ChatGPT, ask it to write queries from natural language prompts, then evaluate what it gives me. Accept, reject, or modify.

I gave ChatGPT the full ReEntry Tech schema and said: "I'm going to give you plain English descriptions of what I need from the database. Write the SQL query for each one. Use my exact table and column names."

Prompt 1: "Show me all clients assigned to case worker #5"

AI wrote
SELECT users.name, users.email
FROM assignments
JOIN users ON assignments.client_id = users.id
WHERE assignments.caseworker_id = 5;
Decision: Accept

This is exactly right. Starts in assignments, filters to caseworker 5, JOINs to users to get the client's name and email. Clean. I would have written this the same way after Exercise 2.

Prompt 2: "How many steps has user #12 completed in module #2?"

AI wrote
SELECT COUNT(*) AS completed_steps
FROM progress
JOIN steps ON progress.step_id = steps.id
WHERE progress.user_id = 12
  AND steps.module_id = 2
  AND progress.completed = true;
Decision: Accept

Correct. Joins progress to steps to know which module each step belongs to, filters to user 12, module 2, completed only. The AS completed_steps alias is a nice touch. Makes the output column name readable instead of just COUNT(*).

Prompt 3: "Find all users who haven't completed any steps"

AI wrote
SELECT users.name, users.email
FROM users
LEFT JOIN progress ON users.id = progress.user_id
  AND progress.completed = true
WHERE users.role = 'citizen'
  AND progress.id IS NULL;
Decision: Modify

The logic is right but I had to stare at it for a while. LEFT JOIN means "include users even if they have no matching progress rows." The AND progress.completed = true in the JOIN condition (not the WHERE) is subtle. Then progress.id IS NULL catches users where the join found nothing.

It works, but it's doing too much in one query for someone reading this the first time. I'd split it into a subquery version that's easier to follow:

SELECT name, email
FROM users
WHERE role = 'citizen'
  AND id NOT IN (
    SELECT DISTINCT user_id
    FROM progress
    WHERE completed = true
  );

Same result. "Get all citizens whose ID doesn't appear in the completed-progress list." The subquery reads like a sentence. The LEFT JOIN version is probably faster at scale but readability matters more when I'm learning.

Prompt 4: "Delete all progress data for a user who's deleting their account"

AI wrote
DELETE FROM progress WHERE user_id = 7;
DELETE FROM questions WHERE user_id = 7;
DELETE FROM assignments WHERE client_id = 7;
DELETE FROM resources WHERE created_by = 7;
DELETE FROM users WHERE id = 7;
Decision: Reject

I said "delete progress data" and it gave me a full cascading hard delete of everything, including the user row itself. That's not what I asked for and it's not what I decided yesterday. My Day 3 edge case decision was soft delete for the user, with optional hard delete of questions only. The AI ignored my design decisions and went nuclear.

What I'd actually write:

-- Soft-delete the user
UPDATE users
SET is_active = false, updated_at = '2026-05-21'
WHERE id = 7;

-- Hard-delete their AI chat history (their choice)
DELETE FROM questions WHERE user_id = 7;

Progress stays (for aggregate reporting). Assignments stay (case worker records). Resources stay (they're shared). Only the personal Q&A gets wiped if the user requests it. Context matters. The AI had no context about my specific design decisions.

Prompt 5: "Get the 5 most recently asked questions across all users"

AI wrote
SELECT questions.body, questions.ai_response, users.name, questions.created_at
FROM questions
JOIN users ON questions.user_id = users.id
ORDER BY questions.created_at DESC
LIMIT 5;
Decision: Accept

Straightforward. JOIN gets the user's name, ORDER BY DESC puts newest first, LIMIT caps it at 5. I hadn't seen LIMIT before but it does exactly what it sounds like. This would power an "admin recent activity" panel.

Takeaway

The AI is good at translating English to SQL syntax. It nailed 3 out of 5 on the first try. But it doesn't know my design decisions. When I said "delete" it heard the SQL keyword, not the soft-delete policy I wrote yesterday. The schema review on Day 3 was the same: the AI is a syntax machine, not a design partner. I have to bring the context. The accept/reject/modify framework is doing real work here because it forces me to evaluate instead of copy-paste.
