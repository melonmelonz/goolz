INSERT, UPDATE, DELETE

Exercise 3 / 5 . Level 2 . 30 min

SELECT was safe. Nothing moves, nothing breaks. Now the database changes. Every query here creates, modifies, or removes a row. These are the operations that matter because they're the ones you can't undo by accident.

INSERT -- creating new rows

Scenario: A new returning citizen signs up 
INSERT INTO users (name, email, role, created_at)
VALUES ('Darius Jackson', 'darius.j@email.com', 'citizen', '2026-05-21');

One new row in users. The id auto-increments so I don't specify it. role defaults to 'citizen' but I'm explicit here because that's what the assignment template said. updated_at starts as NULL since nothing's been edited yet.

Scenario: A case worker adds a new resource 
INSERT INTO resources (title, url, description, category, created_by, created_at)
VALUES (
  'County Transit App',
  'https://transit.county.gov/app',
  'Free bus route planner and real-time arrival tracker for county buses',
  'transportation',
  5,
  '2026-05-21'
);

created_by = 5 is the case worker's user ID. That foreign key tracks who added it. If someone later asks "who put this resource in?", the answer is one query away.

Scenario: A user completes a step 
INSERT INTO progress (user_id, step_id, completed, completed_at)
VALUES (7, 14, true, '2026-05-21');

User 7 completed step 14. One row. The UNIQUE constraint on (user_id, step_id) from yesterday's edge case work means this INSERT will fail if user 7 already has a progress row for step 14. That's intentional. You don't double-count completions.

UPDATE -- modifying existing rows

Scenario: A resource URL changed 
UPDATE resources
SET url = 'https://newportal.county.gov/benefits',
    updated_at = '2026-05-21'
WHERE id = 3;

Only changes the row where id = 3. The WHERE clause is critical. Without it, every single resource in the table gets the same URL. That's the most dangerous thing about UPDATE: forget the WHERE and you corrupt your entire table. The instructor called this "the naked UPDATE problem."

Scenario: Mark a step as completed (update existing progress row) 
UPDATE progress
SET completed = true,
    completed_at = '2026-05-21'
WHERE user_id = 7
  AND step_id = 14;

If the progress row was created when the user first opened the step (with completed = false), this flips it. Two conditions in the WHERE because I need to match the specific user-step combination.

Scenario: A user changes their email 
UPDATE users
SET email = 'darius.jackson@newemail.com',
    updated_at = '2026-05-21'
WHERE id = 12;

Updates one column, leaves everything else alone. The UNIQUE constraint on email means this will fail if someone else already has that address. Good. That's the database protecting itself.

DELETE -- removing rows

Scenario: Remove a glossary term that was a duplicate 
DELETE FROM glossary_terms
WHERE id = 8;

This one is safe because glossary_terms has no foreign keys pointing to it. Nothing else references it. Deleting a term doesn't break any other table. Compare that to deleting a user, which would orphan rows in progress, questions, assignments, and resources.

Scenario: Soft-delete a user (the approach from Day 3) 
UPDATE users
SET is_active = false,
    updated_at = '2026-05-21'
WHERE id = 7;

Not a DELETE at all. It's an UPDATE. This is the soft delete from yesterday's edge case exercise. The row stays, the data stays, but the app knows to treat this user as gone. The WHERE makes me target a specific user. And this doesn't cascade to anything because no rows are removed.

The pattern I see

INSERT is the safest write operation. You're adding, not touching existing data. UPDATE is the most dangerous because a missing WHERE corrupts everything silently. DELETE is permanent (unless you soft-delete). And soft delete is really just UPDATE in disguise. The instructor's rule: "write the WHERE clause first, then write the rest of the query around it." That way you never accidentally hit every row.
