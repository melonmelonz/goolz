Query Plan for a Feature

Exercise 5 / 5 . Level 3 . 30 min

One user story. Every SQL query it needs to work. This is the exercise where everything from the week connects: the user story from Day 1, the API endpoints from Day 2, the schema from Day 3, and the queries from today.

The user story

From Day 1

"As a case worker, I want to see which of my clients are stuck so I can follow up with them."

What "stuck" means

A client is stuck if they have an assigned module with incomplete steps. Specifically: there's a step in the module where the progress row either doesn't exist (never started) or has completed = false. The case worker needs to see: which client, which module, which step, and how long it's been.

The query plan

Step 1: Get the case worker's assigned clients

SELECT client_id, module_id, assigned_at
FROM assignments
WHERE caseworker_id = 5;
What this returns

A list of (client, module) pairs. Say: client 7 assigned to module 1, client 12 assigned to modules 1 and 3, client 19 assigned to module 2. This is the starting point. Everything else fans out from here.

Step 2: For each client-module pair, get the steps and their completion status

SELECT
  steps.id AS step_id,
  steps.title AS step_title,
  steps.step_order,
  progress.completed,
  progress.completed_at
FROM steps
LEFT JOIN progress
  ON progress.step_id = steps.id
  AND progress.user_id = 7
WHERE steps.module_id = 1
ORDER BY steps.step_order;
Why LEFT JOIN

LEFT JOIN instead of regular JOIN because I need to see steps even if the user hasn't started them. A regular JOIN would hide any step that has no progress row. With LEFT JOIN, those steps show up with completed = NULL and completed_at = NULL. That's exactly how I spot "stuck": the step exists but the progress doesn't.

Reading the results

Steps 1-3: done. Step 4: started but not completed (completed = false). Steps 5-6: never started (completed = NULL, no progress row exists). Client 7 is stuck on step 4, "Choose a username." The case worker now knows exactly where to intervene.

Step 3: Get the client's name for the dashboard display

SELECT name, email
FROM users
WHERE id = 7;
Why this is separate

I could have JOINed users into Step 2, but the case worker dashboard probably shows the client's name once at the top, not repeated on every step row. Keeping it separate is cleaner for the front-end to consume. One query gets the person, another gets their step status.

Step 4: How the API serves this

Connecting to Day 2

The Day 2 API endpoint for this was:

GET /api/caseworker/5/clients

The back-end receives this request and runs queries 1, 2, and 3. It packages the results into JSON:

{
  "clients": [
    {
      "name": "Darius Jackson",
      "email": "darius.j@email.com",
      "module": "Setting Up Email",
      "total_steps": 6,
      "completed_steps": 3,
      "stuck_on": "Choose a username",
      "last_activity": "2026-05-16"
    }
  ]
}

The front-end gets this JSON and renders the dashboard. The user story is fulfilled. The SQL is invisible to the case worker. They just see a list of clients with progress bars and stuck indicators.

What I learned

A feature is never just one query. The "stuck clients" dashboard needs three queries minimum, touching four tables, and the results have to be stitched together into a shape the front-end can render. This is why schema design matters. If I didn't have the assignments table linking case workers to clients, or if steps didn't have step_order, or if progress didn't exist as its own table, this whole chain falls apart. Everything I built this week is load-bearing. Nothing was hypothetical.
