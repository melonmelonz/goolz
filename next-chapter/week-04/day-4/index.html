Day 4 -- CRUD & SQL: Making the Database Do Things

NCD . WK4 . D4 . five exercises loaded

Yesterday I designed the tables. Eight of them, with primary keys, foreign keys, constraints, the whole thing. But a schema sitting there is like a filing cabinet with empty drawers. Today: the operations. Create, Read, Update, Delete. Four verbs that cover basically everything a database ever does. And the language that speaks those verbs is SQL.

Morning Warm-up -- CRUD in the Wild

App I picked: Instagram.

Create: posting a photo. A new row lands in some posts table with my user ID, the image URL, the caption, timestamp, location tag if I added one. That row didn't exist before I hit Share.

Read: scrolling the feed. Every post I see is a SELECT query pulling rows from the posts table, probably JOINed with the users table to get the username and profile pic. The feed is just a big read operation with sorting and filtering.

Update: editing a caption after posting. The row already exists. The app sends an UPDATE that changes the caption column for that specific post ID. Everything else in the row stays the same.

Delete: deleting a post. The row gets removed (or soft-deleted, knowing Instagram they probably keep it). But from my perspective it's gone. All the comments and likes that referenced that post ID are now orphans unless they cascade.

Standup

What I did yesterday: designed the full database schema for ReEntry Tech. Eight tables, foreign keys, unique constraints. Got AI feedback, did peer review, handled edge cases like user deletion (soft delete) and duplicate emails (UNIQUE constraint).

What I'm doing today: writing actual SQL queries against that schema. SELECT to read, INSERT to create, UPDATE to modify, DELETE to remove. Turning the conceptual design into operations.

Where I'm stuck: I've never written SQL before today. I understand the concepts from yesterday but translating "get all progress for user 3" into actual syntax is the gap I need to close.

Verbal Mini-Demo

Walked my partner through the case worker dashboard query chain. I said: "The case worker logs in. First query: SELECT client_id, module_id FROM assignments WHERE caseworker_id = 5. That gives me a list of who's assigned to what. For each client-module pair, I need the steps: SELECT id, title FROM steps WHERE module_id = 3 ORDER BY step_order. Then for each step, check progress: SELECT completed FROM progress WHERE user_id = 12 AND step_id = 7. If completed is false or the row doesn't exist, that's where the client is stuck."

Partner asked: "Isn't that a lot of queries? Like one per step per client?" Yes. In production you'd use JOINs to collapse that into fewer queries. But for understanding what's happening, breaking it into individual SELECTs makes the logic visible. Optimize later. Understand first.

Exit Ticket

The thing that clicked today: SQL is just four verbs wearing different hats. SELECT reads, INSERT creates, UPDATE modifies, DELETE removes. Everything else (WHERE, JOIN, ORDER BY) is just narrowing down which rows you're talking about. The hard part isn't the syntax. The hard part is knowing which table to start from and which foreign key to follow. Yesterday's schema work made today possible. Without knowing the relationships, I'd be guessing at JOINs.
