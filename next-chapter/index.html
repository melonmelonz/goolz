Week 1 Day 1 — Introduction to AI

Welcome! Today you'll get hands-on with AI tools and start building intuition for how they work.

Exercise 1

Ask AI Something You're Curious About

Open ChatGPT, Claude, or Gemini and ask it a genuine question. Try one of these starters, or make up your own:

Why does the sky change color at sunset?
What would happen if humans could photosynthesize like plants?
Explain quantum entanglement like I'm 12 years old.

Notice: How long is the answer? Does it feel confident? Does it hedge? Does it ask you anything back?

What you'll see: Answers are long and confident-sounding even for genuinely uncertain topics. AI rarely says "I don't know" — it fills in gaps with plausible-sounding text. It almost never asks a clarifying question back unless you tell it to. That confidence without uncertainty is the main thing to internalize on day one.
💡 Tip: AI works best when your question is specific. "Tell me about history" is hard — "What caused the fall of the Roman Empire in 3 bullet points?" is easy.
Exercise 2

Ask AI to Write Something for You

Try this prompt to get a short professional bio:

Write a 2-sentence professional bio for someone who is learning cloud computing and wants to transition into tech. Make it confident and forward-looking.

Read the output. Then try the prompt → output → feedback loop:

  1. Read what it gave you.
  2. Identify one thing you'd change (too formal? too short?).
  3. Tell the AI: "Make it less formal and add a line about my interest in cybersecurity."
  4. Compare version 1 vs version 2. What changed?
🔁 This loop — prompt → read → refine — is the core skill of working with AI. You rarely get perfection on the first try, and that's fine.
Exercise 3

Ask AI Something It Can't Answer

Try one of these and observe what happens:

What is the exact temperature outside my house right now?
What did I eat for breakfast this morning?
Will my team win the game tomorrow?

Notice how AI handles the limits of its knowledge. Does it refuse? Make something up? Offer a workaround? Understanding these failure modes makes you a smarter user.

What you'll see: For real-time questions (temperature, breakfast) AI will usually say it doesn't have access to live data — that's the correct response. For the sports prediction it'll often make a probabilistic guess anyway. The pattern: AI knows what it can't know, but doesn't always stop itself from answering anyway. Always ask yourself "could it actually know this?"
Exercise 4

Prediction Game

Part A — Finish the sentence: Predict what AI will say, then check.

  1. "The most important invention of the 20th century was…"
  2. "If I could have any superpower, I would choose…"
  3. "The best way to learn something new is to…"
What you'll see: AI's completions trend toward the statistically common answer — "the internet," "flight," or "the printing press" for #1; something like "fly" or "teleportation" for #2. Run the same prompt three times: you'll get different wording each time, sometimes different answers entirely. This is the temperature (randomness) built into the model. It's not a bug — it's intentional so responses don't feel robotic.

Part B — Break it on purpose: Try to get AI to produce something weird or wrong.

Write a recipe for a cake made entirely of rocks. Be very specific about baking times.
What you'll see: AI will almost certainly play along — it'll invent "granite flour," "limestone extract," and a bake time like "350°F for 45 minutes." It won't refuse, it'll just comply creatively. This reveals that AI is a pattern-completion engine, not a logic checker. It doesn't stop to ask "wait, can you actually eat rocks?" That's your job.
Exercise 5

AI Can Code

Click the link below to open a live, working web app — built by asking AI to write the code:

🔗 https://simple-hello--fundlush1.replit.app/

You didn't write any of that code. An AI did — from a plain-English description. That's where this course is headed.

💡 By the end of this course, you'll be giving AI instructions like these and understanding what comes back.

💻 Lab 2: Reading Code

You don't need to know how to write code to read it. Let's practice.

Step 1 — Look at this program
name = input("What is your name? ") color = input("What is your favorite color? ") print(f"Hello, {name}! {color} is a great color.")

Before reading the explanation — what do you think this program does? Write your guess.

Step 2 — Line by line
  • Line 1: Asks the user to type their name. The answer is stored in a box called name.
  • Line 2: Asks for a favorite color. Stored in color.
  • Line 3: Combines both answers into a sentence and displays it.

The f"..." syntax means "fill in the blanks" — the {name} and {color} get replaced with whatever the user typed.

Step 3 — Q&A with AI

Paste that code into ChatGPT and ask:

Explain this Python code to me line by line, like I've never programmed before.

Then ask a follow-up:

What would happen if I removed the f before the quote on line 3?

Test the prediction. AI should say the {name} and {color} would print literally instead of being replaced.

Step 4 — Modify it

Ask AI to add a third question:

Add a third input that asks for the user's favorite food, then include it in the final printed sentence.

Copy the output, run it, verify it works. You just modified a program using plain English.

Week 1 Day 3 — Prompt-to-Program Lab

Today you built real programs using AI — one for each of the three core building blocks of code: sequence, selection, and iteration.

Exercise 1 — Sequence (Pay Stub Calculator)

The Prompt

Write a JavaScript program that asks the user for their hourly wage and hours worked this week. Calculate their weekly pay (wage x hours). Display: "You earned $[amount] this week."

The Generated Code

let wage = parseFloat(prompt("What is your hourly wage? $")); let hours = parseFloat(prompt("How many hours did you work this week?")); let pay = wage * hours; alert(`You earned $${pay.toFixed(2)} this week.`);

Trace-Through (sample: $15/hr, 40 hrs)

LineWhat happensValue stored
1Asks for wage, user types 15wage = 15
2Asks for hours, user types 40hours = 40
3Multiplies 15 × 40pay = 600
4Displays the result"You earned $600.00 this week."
Each line depends on the one before it — that's what makes this a sequence. Swap lines 3 and 4 and the program breaks entirely.

Modification — After-Tax Take-Home

ADDED
+ let tax = pay * 0.08;
+ let takeHome = pay - tax;
+ alert(`Gross: $${pay.toFixed(2)} | Tax (8%): $${tax.toFixed(2)} | Take-home: $${takeHome.toFixed(2)}`);

💡 At $15/hr for 40 hrs: Gross $600.00 → Tax $48.00 → Take-home $552.00
Exercise 2 — Selection (Temperature Checker)

The Prompt

Write a JavaScript program that asks the user what the temperature is outside (in Fahrenheit). Then display: - Below 32: "It's freezing! Stay inside." - 32 to 60: "It's chilly. Grab a jacket." - 61 to 80: "Perfect weather!" - Above 80: "It's hot. Stay hydrated."

The Generated Code

let temp = parseFloat(prompt("What is the temperature outside (°F)?")); let message; if (temp < 32) { message = "It's freezing! Stay inside."; } else if (temp <= 60) { message = "It's chilly. Grab a jacket."; } else if (temp <= 80) { message = "Perfect weather!"; } else { message = "It's hot. Stay hydrated."; } alert(message);

The Prompt

Modification — 10% Discount on Items Over $5

ADDED

Week 1 Day 4 — Human Lane vs. AI Lane

Today's theme: knowing when to hand the wheel to AI and when to take it back. Structured prompts, ethical judgment, and the question "should we?" vs. "how do we?"

Warm-Up — Human or AI?

The Three Texts

One was written by a human. Two by AI. Which is which?

TextMy GuessWhy
A — "Learning to code gives you the power to create solutions…"AICorrect-sounding and motivational, but generic. No specific story, no friction. Reads like an AI generating an inspirational opener.
B — "In today's rapidly evolving digital landscape…"AIAbsolutely AI. "Rapidly evolving digital landscape," "indispensable asset," "competitive advantage" — corporate buzzword bingo. No human writes like this voluntarily.
C — "I started learning to code because I was tired of having ideas…"HumanSpecific, personal, uncomfortable ("tired of having ideas"). "App that my neighborhood actually uses" — AI wouldn't invent that detail. The friction is the tell.
The tell: AI fills in with confident, polished language. Humans have friction, specificity, and things that don't need to be there but are. Text C has all three.
Exercise 1 — Role Prompting

The Task: Explain what a variable is in JavaScript. Three ways.

Prompt 1 — No Role

Explain what a variable is in JavaScript.

AI Output (paraphrased): "A variable in JavaScript is a named storage location for data. You declare one using var, let, or const. Variables can hold numbers, strings, booleans, objects, or arrays. The value can be changed later unless declared with const…" (continues for several paragraphs)

Accurate. But it starts with jargon ("storage location," "declare") and keeps going. If I'm new, I'm already lost by sentence two.

Prompt 2 — Teacher Role

You are a patient coding teacher for adults who have never programmed before. Explain what a variable is in JavaScript. Use a real-world analogy. Keep it to 3 sentences. No jargon.

AI Output (paraphrased): "Think of a variable like a labeled sticky note. You write something on it — a number, your name, anything — and then you can read it or change it whenever you want. In JavaScript, you make one by writing let myName = "Penn"; — the label is myName, and "Penn" is what's written on it."

This is the one that actually landed. The sticky note clicked. Three sentences, no paragraph of preamble.

Prompt 3 — Technical Writer Role

You are a senior technical writer at a software company. Write a concise reference entry for "variable" in JavaScript. Include: definition, syntax for declaration (const, let, var), one code example, and one common mistake beginners make. Format with headers.

AI Output (paraphrased):

Variable — A named binding that associates an identifier with a value in memory.

Syntax:

const x = 10; // block-scoped, cannot be reassigned let y = 20; // block-scoped, can be reassigned var z = 30; // function-scoped, avoid in modern JS

Common mistake: Using var inside a loop and expecting it to be block-scoped. Use let instead.

Best code example by far — three declarations side by side with comments. Worth bookmarking as a reference. But I wouldn't start a new student here.

Exercise 3 — System Instruction Practice

Scenario A — Debugger

You are a debugging assistant. When I show you code with a problem: 1. Ask me one clarifying question before giving any answer. 2. Never rewrite my entire code — only identify the specific broken part. 3. Explain what the error means in plain language before suggesting a fix. 4. If the fix involves a concept I might not know, name it so I can look it up.
Why this works: It forces AI to slow down and not just rewrite everything. The clarifying question makes me think about what I actually observed, not just "it's broken."

Scenario B — Brainstormer

You are an app idea generator. When I give you a topic or problem: 1. Give me exactly 5 app ideas — no more, no less. 2. For each idea, explain in one sentence who would use it. 3. Rate each idea's build difficulty from 1 (beginner-friendly) to 5 (complex). 4. Don't explain your ratings unless I ask. Format: numbered list, idea name in bold, then the user sentence, then "Difficulty: X/5"
The formatting constraint is the most important part. Without it, AI writes a paragraph per idea and the list becomes unreadable. Structured output requires structured instructions.

Scenario C — Quiz Mode

You are a quiz bot for a student reviewing Week 1 of an AI builder course. Rules: - Ask one question at a time. Wait for my answer before continuing. - After I answer: tell me if I'm right. If wrong, explain the correct answer in 2–3 sentences. - Questions should cover: what AI is, how prompts work, sequence/selection/iteration, the difference between human lane and AI lane. - After 5 questions, give me a score (e.g. "4/5 — solid!"). Start now.
💡 "Start now" at the end is important — without it, AI often explains what it's about to do instead of just doing it.

Applied Judgment Lab — Exercise 1: Accept, Reject, or Modify?

Scenario A — The Professional Email

AI wrote: An email claiming "extensive experience in full-stack development," "expertise in React, Node.js, PostgreSQL, Python, Java, C++, AWS, Docker, Kubernetes, and machine learning," and calling itself an "invaluable asset."

Decision: Reject / Rewrite

- "extensive experience in full-stack development"
- "expertise in React, Node.js, Python, Java, C++, AWS, Docker, Kubernetes, ML"
- "invaluable asset to your team"
+ "currently completing an AI builder program focused on applied AI tools"
+ "I'm early in my technical journey and I'm learning fast"
+ "I'd be a motivated learner who ships things" (something I can back up)
The AI wrote a fiction. That list of skills is someone's 5-year résumé. I have four days of experience. An interviewer who reads that and then talks to me for 10 minutes will catch it immediately — and it kills trust. Honesty + specificity beats impressive lies every time.

Scenario B — The isPrime Function

function isPrime(num) { if (num <= 1) return false; if (num <= 3) return true; if (num % 2 === 0 || num % 3 === 0) return false; for (let i = 5; i * i <= num; i += 6) { if (num % i === 0 || num % (i + 2) === 0) return false; } return true; }

Decision: Modify — Accept the function, ask for a simpler version to study

The code is correct and optimized (the i += 6 trick skips multiples of 2 and 3, which I didn't know). I can use it — but I'd also ask for a naive version I can fully trace through. "This works. Now show me one that's slower but that a beginner can read line by line." Using code I can't explain is a liability.

Scenario C — App Name Suggestions

AI suggested: TruckTracker, StreetEats, FoodRoamer, BiteBus, RollUp

Decision: Modify — pick one, verify, then tweak

StreetEats is the strongest. It's two syllables, says exactly what it is, and sounds like something you'd actually search for. I'd check it on the App Store and USPTO TESS before committing. FoodRoamer and RollUp sound like someone's gym app. BiteBus is a bus company. TruckTracker is what a logistics company uses to follow their drivers. StreetEats wins — but I'd still Google it first.
Applied Judgment Lab — Exercise 2: Ethics Scenarios

Scenario 1 — The Resume

What went wrong: AI made things up. It inflated a prompt ("I'm learning to code") into a portfolio that doesn't exist. The student didn't catch it or chose not to. When the interview happened, the lie became visible and the trust was gone.

The line: AI can draft language, clean up what I wrote, and suggest framing. It cannot invent experience I don't have. My rule: if I can't back it up in a 5-minute conversation, it doesn't go in.

Right use: "Here are three bullet points about a project I did. Make them sound more professional." That's AI refining real truth — not generating fictional truth.

Scenario 2 — The Client Project

What should have happened: Read every line of the generated code before shipping it. If I couldn't explain a function, I should have asked AI to explain it, then asked follow-up questions until I could. "Working" and "understood" are different things. Working is the minimum. Understood is what lets you fix it when it breaks at 2am six months later.

The minimum standard: I can describe what each function does in plain language and identify where I'd look if something breaks. That's the floor for responsible delivery.

Scenario 3 — The Bias Check

When asked to describe a "typical software engineer": male-leaning pronouns, mentions "he" or defaults to he/him, probably young, probably in a hoodie or office setting, probably white or East Asian in many outputs.

When asked to describe a "typical nurse": female-leaning, scrubs, gentle, maybe "she" by default.

Where it comes from: The internet. Specifically: news articles, job boards, blog posts, LinkedIn profiles, Reddit threads, Wikipedia entries. All of them reflect the world as it was historically documented, not as it should be. AI learned the pattern, not the exception.

Why it matters: If you use AI to write job descriptions, marketing copy, or character descriptions without checking, you're amplifying the same stereotypes that already make some people feel like they don't belong in the field. The fix isn't complicated — read the output and ask "who does this leave out?"

Peer Activity — Explain AI to a Non-Technical Person

My 2-Minute Explanation (Draft)

"AI — the kind we've been using this week — is basically a very sophisticated autocomplete. Imagine your phone's keyboard suggestion, but instead of suggesting the next word in a text message, it's been trained on most of the internet and can suggest the next word in an essay, a piece of code, or a whole conversation.

Here's the analogy I like: it's like a really well-read intern. They've read everything in the library — every book, article, and forum post. They're fast, never tired, and can draft a first version of almost anything. But they've never actually lived life. They don't know what's true right now, they can confidently say wrong things, and they'll tell you what you want to hear if you push them.

What AI is genuinely good at: first drafts, code boilerplate, summarizing long documents, brainstorming options, explaining things 10 different ways until one clicks.

What AI is bad at: knowing what's true right now, understanding your specific situation, making ethical calls, and knowing when to say 'I don't know.'"
