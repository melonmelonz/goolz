Day 1 -- Next.js Intro: React Plus a Server

NCD . WK6 . D1 . five exercises loaded . react gets a backend

Last week I shipped a Cloudflare Pages site with serverless functions and KV, all in plain Preact with no build step. That worked because the project was small and the backend was two API endpoints. This week React gets a server of its own. Next.js. The framework that Netflix, TikTok, Notion, and the learning platform we are using right now run on. When job postings say "React developer" they increasingly mean "Next.js developer," which I take at face value because the companies hiring are the ones writing the postings.

What Next.js adds is not complicated to list: file-based routing, server-side rendering, API routes, layouts that persist across navigation, and a deployment story that is one command. What is actually new to think about is the server/client split. Every component is a server component by default. It runs on the server, sends HTML to the browser, and the browser never sees the JavaScript. If you need useState or onClick or anything interactive, you mark the file "use client" and it runs in the browser. The discipline is: start on the server, opt into the client only when you need interactivity. Keep client components small and focused.

I have been living in the Cloudflare Workers world where everything runs at the edge by default. Next.js server components are the same idea applied to React. The server does the work, the browser gets the result. The difference is that Next.js gives you a framework opinion about where the boundary is, and the "use client" directive is how you declare it. In Workers I make that choice by deciding what goes in a Function vs. what ships as static HTML. Here the framework makes it structural.

Lecture Notes -- File-Based Routing

The folder structure IS the routing. No router library, no config file.

app/
  page.js          -->  /          (home page)
  about/
    page.js        -->  /about
  contact/
    page.js        -->  /contact
  blog/
    page.js        -->  /blog
    [id]/
      page.js      -->  /blog/1, /blog/2, /blog/any-id

Create a folder, put a page.js in it, the route exists. The [id] with square brackets is a dynamic route; it matches any value and hands it to your component as a parameter.

Lecture Notes -- Layouts

layout.js wraps every page. Navigation, footer, overall structure. When users navigate, the layout stays and only the page content swaps. {children} is where the current page gets inserted.

// app/layout.js
export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>
        <nav>...</nav>
        <main>{children}</main>
        <footer>...</footer>
      </body>
    </html>
  );
}

This is the pattern I have been doing by hand on every project. On goolz.org the Win95 desktop chrome is a layout. On mute-pixel the starfield + header + footer is a layout. Next.js makes it structural instead of manual.

Lecture Notes -- Server vs. Client Components

Server components (the default): run on the server, send finished HTML, can access the database directly, cannot use useState, useEffect, or event handlers. Fast because the browser gets ready-to-display HTML.

Client components (opt-in): run in the browser, needed for interactivity. Add "use client" at the top of the file. Can use useState, useEffect, onClick, onChange.

// SERVER component (default, no directive)
export default function WelcomeBanner() {
  return <h1>Welcome to the app!</h1>;
}

// CLIENT component (note the directive)
"use client";
import { useState } from "react";
export default function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

Rule of thumb: start with server components. Only add "use client" when you need interactivity. Keep client components small: a button, a form, a toggle. Not entire pages.

Convention for file placement: shared client components go in app/components/. Route-specific components (a form that lives only on the recipes page) get colocated inside that route folder.

Verbal Mini-Demo -- Server vs. Client Components

Partner asked me to explain the split using my Counter component. Page.js on the home route is a server component. It runs on the server, renders to HTML, ships to the browser. The browser never sees the JSX or the import statements. It gets finished HTML. Inside that page I import Counter, which has "use client" at the top. That directive is the boundary. Everything in Counter.jsx ships to the browser as JavaScript. The browser hydrates it, wires up the onClick handlers, and useState works because it is running in a browser context.

Without "use client", Next.js tries to run Counter on the server. The server hits useState, which does not exist in a server context, and throws. The error message is explicit: "You're importing a component that needs useState. It only works in a Client Component." The fix is one line at the top of the file. The lesson is that the default is the server, and you opt into the client. Not the other way around.

Follow-up question from partner: "Can you nest a server component inside a client component?" Short answer: not directly. A client component can render {children} passed from a server parent, but it cannot import and render a server component. The boundary only goes one direction. Server can import client. Client cannot import server. This matters for keeping the client bundle small.

The Thing That Clicked Today

File-based routing is not a feature. It is an opinion about where decisions should live. In plain React I install react-router, create a routes config, map paths to components, and maintain that mapping as the app grows. In Next.js I create a folder. The decision about what URL a page lives at is made by the file system, not by a config file I have to keep in sync. The same thing is true of layouts. I do not configure a layout provider. I create a layout.js file in a directory and everything below that directory gets wrapped.

This is the same pattern I saw on Friday with Cloudflare Pages Functions. Put a JavaScript file in functions/api/ and the API route exists. No router, no config, no registration. The file system is the source of truth for routing. Next.js and Cloudflare Pages arrived at the same conclusion from different directions. I think this is where web development is heading: fewer config files, more convention. The structure IS the config.
