Primer · three ways to attach CSS

Same paint, different brushes. There are three places CSS can live:

  • Inline — on the element itself: <h1 style="color: navy">. Quick test, but messy and not reusable. Avoid in real code.
  • Internal — in a <style> block in the page's <head>. Fine for one tiny page or a single-file demo (this lesson uses internal under the hood).
  • External — in its own .css file, linked from the <head>: <link rel="stylesheet" href="styles.css">. This is the way. Cached by the browser. Reusable across pages. Step 1 below shows the link.

Gotcha: type colr: blue by accident and the browser silently ignores the rule — no error, no warning, just nothing happens. CSS forgives spelling mistakes. Always check spelling and the missing semicolon first.

Primer · the box model

Every element on a page is a rectangle. That rectangle has four layers, from inside out:

+-------------------------------+
|           MARGIN              |   ← space OUTSIDE the border
|  +-------------------------+  |     (pushes neighbors away)
|  |        BORDER           |  |
|  |  +-------------------+  |  |   ← the visible edge
|  |  |     PADDING       |  |  |
|  |  |  +-------------+  |  |  |   ← space INSIDE the border
|  |  |  |   CONTENT   |  |  |  |     (between border and content)
|  |  |  +-------------+  |  |  |
|  |  +-------------------+  |  |   ← your text, image, child elements
|  +-------------------------+  |
+-------------------------------+
  • Padding = breathing room inside the box (between border and content).
  • Margin = breathing room outside the box (between this box and its neighbors).
  • Border = the line drawn at the edge.

Mnemonic: more space inside an element? Padding. More space between elements? Margin. The DevTools "Computed" tab paints this exact diagram in real numbers — show it to the class.

1 Tell HTML where the CSS lives

CSS usually lives in its own file (e.g. styles.css). You connect it to your HTML with one line inside the <head>:

<link rel="stylesheet" href="styles.css">

Then anything you write in styles.css applies to your page. As a tiny first taste, let's make the page readable — set a sane font and add some breathing room:

body {
  font-family: Arial, sans-serif;
  line-height: 1.6;
  padding: 20px;
  max-width: 640px;
  margin: 0 auto;
}

Hint: body means "every <body> element" — there's only one, so this styles the whole page. margin: 0 auto is the classic centering trick.

2 Element selectors — paint by tag name

A CSS rule has two parts: a selector (who) and a declaration block (how). Here we pick every <h1> and every <h2> on the page and color them:

h1 {
  color: navy;
}

h2 {
  color: teal;
  border-bottom: 2px solid teal;
  padding-bottom: 2px;
}

Selector cheat-sheet: h1 = every tag, #some-id = the one element with that id, .some-class = every element with that class. We'll use a class in step 5.

3 Backgrounds & colors

Color isn't just for text. Anything visual has a background too. Let's give the page some warmth and turn the title into a banner:

body {
  background: #fffaf0;  /* warm off-white */
}

h1 {
  background: navy;
  color: white;
  padding: 10px 14px;
}

Colors can be names (navy), hex (#fffaf0), or rgb(). Pick what reads cleanest. Modern CSS also has hsl() and oklch() for fancier color work.

or press 3
4 Fonts — change how text feels

One font-family sets the default for everything. You can override it per element. Mixing a serif body with a sans-serif heading is a classic move:

body {
  font-family: Georgia, serif;
}

h1, h2 {
  font-family: 'Trebuchet MS', sans-serif;
  letter-spacing: 0.5px;
}

List multiple fonts in case the first one isn't installed — the browser falls back down the list. End with a generic family (serif, sans-serif, monospace) so you always get something.

or press 4
5 Flexbox — lay things out in a row (or column)

By default, block elements stack vertically. To put them in a row, give their parent display: flex. Look at the three "friend" cards in the preview — they're stacked. One rule lays them out side by side:

/* The cards live inside <div class="row">…</div> */
.row {
  display: flex;
  gap: 12px;
  justify-content: center;
  align-items: center;
  flex-wrap: wrap;
}

.card {
  background: #ffeb3b;
  padding: 10px 14px;
  border: 2px solid #333;
  border-radius: 6px;
  font-weight: bold;
}

Five flexbox ideas to know:

  • display: flex — children sit on the main axis (a row by default).
  • flex-direction: row | column — flip the main axis to vertical.
  • justify-content — pack items along the main axis: flex-start, center, flex-end, space-between, space-around, space-evenly.
  • align-items — line items up on the cross axis: stretch, center, flex-start, flex-end.
  • gap — space between items. Beats margin every time.

There's also flex-grow and flex-shrink on the children, for "make this one fill the leftover space" tricks. The full guide is linked at the bottom.

6 Hover & transitions — make it feel alive

CSS lets you change styles based on what the user is doing. :hover kicks in when the mouse is over an element. transition smooths the change so it doesn't jump.

.card {
  /* whenever any of these change, animate over 200ms */
  transition: background 200ms ease, box-shadow 200ms ease, transform 200ms ease;
  cursor: pointer;
}

.card:hover {
  background: #ffd24d;
  transform: translateY(-3px);
  box-shadow: 0 4px 0 rgba(0,0,0,0.25);
}

h1 {
  transition: letter-spacing 250ms ease;
}
h1:hover {
  letter-spacing: 3px;
}

After applying, hover the friend cards in the preview — they lift, the color warms, the shadow grows. Hover the <h1> too. The transition property is the magic glue.

The shape of transition: property duration timing-function delay. Common timings: ease, ease-in-out, linear, cubic-bezier(...).

7 @keyframes — full-blown animations BONUS

Transitions only animate between two states (resting → hovered). For more — looping wobbles, multi-stage fades, choreography — you write a @keyframes rule and attach it with animation.

/* 1. Define a named timeline */
@keyframes fade-in {
  from { opacity: 0; transform: translateY(8px); }
  to   { opacity: 1; transform: translateY(0); }
}

@keyframes wobble {
  0%   { transform: rotate(-2deg); }
  50%  { transform: rotate(2deg) translateY(-2px); }
  100% { transform: rotate(-2deg); }
}

/* 2. Attach with the animation shorthand:
   name | duration | timing | delay | iteration | direction | fill */
h1 {
  animation: fade-in 600ms ease-out both;
}

.card {
  animation: wobble 2.5s ease-in-out infinite;
}
.card:nth-child(2) { animation-delay: 0.3s; }
.card:nth-child(3) { animation-delay: 0.6s; }

/* Be a good citizen: respect users who turn motion off. */
@media (prefers-reduced-motion: reduce) {
  * { animation: none !important; }
}

What you'll see when you apply this: the title fades up into place, and the three friend cards do a staggered, gentle wobble forever — each one starts a beat after the last. That's animation-delay + :nth-child() doing the choreography.

Three concepts in one step:

  • @keyframes — a named timeline of styles, percentages from 0% → 100% (or just from/to).
  • animation shorthand — name + duration + timing + delay + how many iterations + direction + fill mode.
  • :nth-child(n) — pseudo-class that matches the Nth element. Combine with animation-delay for staggered effects.

Keep it tasteful: a 300–600ms entrance animation reads as "polished"; a constant infinite wobble on every card reads as "carnival." For UI, restraint wins.

Things to chew on · class discussion

Ask the room. Let people argue. Click Show answer when you're ready.

display: none vs visibility: hidden — they both make an element invisible. Are they the same?

No. They differ in three big ways:

  • Layout: display: none removes the element entirely — surrounding elements collapse into the space. visibility: hidden keeps the box; the space stays reserved.
  • Tab order & screen readers: display: none is gone — not focusable, not announced. visibility: hidden is also skipped by both, but the layout slot is still there.
  • Animations: you can't transition display (well, until very recently — it's still spotty). visibility can transition, and opacity: 0 is usually what you actually want for fade effects.

There's also opacity: 0 — invisible, takes space, AND still focusable/clickable. That's the gotcha.

px vs em vs rem vs % — four ways to say "this size." When do you use each?
  • px — fixed pixels. Predictable, but doesn't scale if a user bumps their browser font size for accessibility. Use for borders, tiny details.
  • em — relative to the current element's font-size. Compounds in nested elements (a 1.2em inside a 1.2em inside a 1.2em… runaway).
  • rem — relative to the root (<html>) font-size. No compounding. The modern default for typography and spacing.
  • % — relative to the parent's matching dimension. Great for fluid widths.

Rule of thumb: rem for spacing & type, % or fr for layout, px for hairlines.

margin vs padding — which one collides with its neighbors, and what is "margin collapse"?

Margin is space outside the border — it pushes neighbors away. Padding is space inside the border — between border and content. Padding never affects neighbors.

Margin collapse: when two vertical margins meet (e.g. one element's margin-bottom: 20px and the next's margin-top: 30px), they don't add to 50 — they collapse into the larger one (30). Catches everyone the first time.

Padding doesn't collapse. Flex/grid gap doesn't collapse. That's a major reason gap beats margin for layout.

Why does width: 100% sometimes overflow its parent?

Default box-sizing is content-box: width is just the content. Padding and border are added on top. So width: 100% + padding: 20px = parent width + 40px → overflow.

Fix it once for the whole project:

*, *::before, *::after { box-sizing: border-box; }

Now width includes padding and border. Most modern stylesheets start with this exact rule.

Is !important a tool or a red flag?

Usually a red flag. It overrides normal specificity rules, which means later you'll need another !important to undo it. Specificity wars escalate fast.

When it's fine: utility classes (.hidden { display: none !important }), user stylesheets overriding sites, beating third-party CSS you can't edit.

When it's a smell: patching your own selectors because you can't figure out why they're losing. The fix is usually a more specific selector, not nuclear-grade override.

I changed the CSS, hit save, refreshed — and nothing happened. What's going on?

This is the most universal CSS pain. Probably one of:

  • Typo in a property. colr: blue, bcolor: red, marign: 8px — CSS silently drops rules it doesn't recognize. No error. Inspect the rule in DevTools — it'll have a yellow warning triangle.
  • Missing semicolon. A missing ; can break the rule below it.
  • Specificity loss. A more specific selector elsewhere is winning. DevTools shows you which rule won and which got crossed out.
  • Browser cache. Hard refresh: Ctrl/Cmd + Shift + R.
  • You're editing the wrong file. It happens. Check the path in your <link>.

Habit to build: after a "nothing happened" moment, open DevTools → Elements → click the element → look at the Styles pane. If your rule isn't there at all, it's a typo or a wrong file. If it's there but crossed out, it's specificity.

Why doesn't text-align: center center a <div>?

Because text-align only centers inline content (text, inline images, inline-block elements) inside the element it's set on. It doesn't center the element itself.

To center a block element horizontally: give it a width and margin-inline: auto (or the classic margin: 0 auto).

To center anything (block or inline) in both directions, the modern answer is one rule on the parent:

.parent {
  display: grid;
  place-items: center;
}

Flexbox version: display: flex; align-items: center; justify-content: center.
