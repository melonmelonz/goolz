PURE CSS.

Five live demos. Zero JavaScript. view-source no JS Open DevTools → Network → JS — the count is 0. Everything below is HTML structure plus stylesheet rules.

Why this exists

CSS used to be "make text pink." Modern CSS can hold state, animate, transform in 3D, react to user input, even paint a canvas — all without a single line of JavaScript. Each demo below shows one trick. Click How does it work? on any demo to read the trick.

1 PaintCSS — drag to paint headliner

Pick a color, then drag your mouse across the canvas. The canvas remembers every cell you've touched. Every pixel of this is CSS — no onclick, no script. Touch note: the trick uses :hover, which mobile browsers don't have a great answer for — tap-and-drag may not paint reliably on phones. Use a mouse if you can.

Clear canvas wipes everything. Reset color puts the picker back on Win95 blue. The Eraser ✖ paints with the canvas color (subtle one-cell-at-a-time clear).
How does it work?

Three CSS tricks stacked together — no JS:

  • The "sticky paint" trick. Each cell has transition: background 0s 99999s. When you hover, a more specific rule overrides it with transition: background 0s 0s and sets the painted color. When you stop hovering, the original transition kicks back in — but its delay is 99999 seconds, so the unpaint never visibly happens. Result: the cell stays painted.
  • The color picker. Sixteen <input type="radio"> elements (one per color) hold the "selected color" state. The labels are the visible swatches.
  • :has() drives the canvas. A rule like .paint:has(#pc-c6:checked) { --paint-color: #ff0000; } reads the selected radio and updates a custom property on the parent. Every cell's hover rule reads var(--paint-color), so changing the picker changes the paint.
.canvas i {
  transition: background 0s 99999s; /* delay = forever */
}
.canvas i:hover {
  background: var(--paint-color);
  transition: background 0s 0s;     /* no delay on the way in */
}
.paint:has(#pc-c6:checked) { --paint-color: #ff0000; }

Clear canvas uses one more trick — a hidden checkbox #pc-clear with a <label> styled as a button. When checked, :has() attaches a @keyframes animation to every cell that fades the background to transparent. While checked, the hover paint rule is also overridden, so painting is paused. Click the button again to uncheck and resume painting.

.paint:has(#pc-clear:checked) .canvas i {
  animation: paint-clear-fade 220ms ease forwards;
}
@keyframes paint-clear-fade { to { background: transparent; } }

Caveats: the cells aren't really "painted" — they just look that way. Reload and the canvas is blank. Eraser works because it just paints with the canvas background color. The classic limitation of clever CSS-only state.

2 Solar system — pure @keyframes

Four planets, four orbits, four loop durations. The whole thing is one shape (a circle) repeated, plus one animation (rotate).

How does it work?

Each "arm" is a zero-size element pinned at the center of the stage. The planet sits offset from that center using left:. Then we rotate the arm — and the planet rides along the circle.

@keyframes orbit {
  from { transform: rotate(0deg); }
  to   { transform: rotate(360deg); }
}

.arm     { position: absolute; top: 50%; left: 50%;
            animation: orbit 7s linear infinite; }
.arm .planet { position: absolute; left: 75px; /* orbit radius */ }

Different orbit radii (left values) and different durations give you the full system. The sun also pulses (a separate @keyframes on its box-shadow) and the stars twinkle (opacity alternation on a background made of radial-gradient dots — yes, that's all CSS too).

How does it work?

Two CSS properties unlock the third dimension:

  • perspective on the parent — sets how far away the camera is. Smaller numbers = more dramatic 3D.
  • transform-style: preserve-3d on the cube — children get to live in 3D space instead of being flattened.

Each face starts on top of the other, then we rotate it and push it outward by half the cube's size:

.cube { transform-style: preserve-3d; animation: spin-cube 14s linear infinite; }

.face-front  { transform: translateZ(80px); }
.face-back   { transform: rotateY(180deg) translateZ(80px); }
.face-right  { transform: rotateY(90deg)  translateZ(80px); }
.face-left   { transform: rotateY(-90deg) translateZ(80px); }
.face-top    { transform: rotateX(90deg)  translateZ(80px); }
.face-bottom { transform: rotateX(-90deg) translateZ(80px); }

Mental model: each face is initially flat-on, then you rotate it to point outward, then push it away from the center by half the cube's edge so the inside surface lines up. Pure geometry; no JS, no canvas, no WebGL.

4 Tabs — radio inputs holding state

Click a tab. The content below updates. No JavaScript reads or writes to the DOM — the radio button is the state, and CSS shows/hides panels based on which one is checked.

Overview. CSS is a rule-based language. You describe the look you want; the browser figures out how to draw it. Every visual feature on this page (and the entire one you're reading) is driven by the same idea.
Selectors tell the browser who. p is every paragraph. .note is every element with class "note." #hero is the one with id "hero." Modern selectors include :has(), :not(), :nth-child(), and many more.
Layout. Flexbox arranges things in a row or column. Grid arranges them in two dimensions at once. Both replaced years of float-and-table-hack. display: flex on a parent is usually enough to fix 80% of layout pain.
Animation. transition smooths a change between two states. @keyframes + animation describes a multi-step timeline that loops. The browser does the math; you describe the destination.
How does it work?

The radio inputs are visually hidden but still focusable. Clicking the matching <label> checks them. Then :has() reads which one is checked and shows the right panel:

.tab-panel { display: none; }

.tabs:has(#tab-1:checked) .panel-1,
.tabs:has(#tab-2:checked) .panel-2,
.tabs:has(#tab-3:checked) .panel-3,
.tabs:has(#tab-4:checked) .panel-4 { display: block; }

Before :has() existed, the same effect was done with the general sibling combinator (~). It still works, just uglier. :has() is the modern way.

5 Defragmenter — staggered keyframes

The classic Win95 defrag screen, animated entirely with @keyframes and animation-delay. Every block runs the same animation — they're just offset in time.

How does it work?

240 blocks all share one @keyframes rule that walks through gray → red → blue → green. Each block gets a different animation-delay using :nth-child() — so they ripple through the cycle instead of all flashing in sync.

@keyframes defrag-cycle {
  0%  { background: #c0c0c0; }   /* fragmented */
  15% { background: #ff4d6d; }   /* bad block found */
  30% { background: #1084d0; }   /* relocating  */
  60% { background: #2cb673; }   /* optimized   */
  100%{ background: #c0c0c0; }
}

.blocks i           { animation: defrag-cycle 12s ease-in-out infinite; }
.blocks i:nth-child(8n+2) { animation-delay: 0.4s; }
.blocks i:nth-child(8n+3) { animation-delay: 0.8s; }
/* …and so on. The wave is just spaced delays. */

The progress bar uses the same trick on a single element: transform: scaleX() animated from 0.05 → 0.95 over the loop, with the bar's left origin so it grows rightward.

CSS techniques featured here
:has() — the parent / "looks-down" selector

The big new thing. Until 2023 it didn't exist. parent:has(child:state) styles the parent based on the child. It's why we can run a color picker without a single onclick.

transition-delay: 99999s — the "stuck" trick

A property changes on hover; the transition has a huge delay before it animates back. Effectively permanent. Used by the paint canvas above.

transform-style: preserve-3d + perspective

Switches the child elements from "flat" to "in 3D space." Combined with rotateX/rotateY/translateZ, you get a real cube — not a fake-looking one.

@keyframes + animation-delay + :nth-child()

Define one timeline, share it across many elements, stagger the start by index. Every "wave," "stagger," or "ripple" effect on the web is this pattern.

Hidden radios + <label> as state

The browser already tracks which radio is "checked." That's free state. Style the labels to look like buttons; CSS reads :checked via :has() or the sibling combinator.

Custom properties (CSS variables)

--paint-color: red on a parent; background: var(--paint-color) on every child. Change the parent, every child updates. Themes and dynamic colors without a single line of script.

The honest part: what CSS still can't do

Pure CSS state is session-only — every demo above resets on reload. There's no localStorage, no math, no fetching data. JavaScript is still the right tool when you need persistence, real interactivity, or anything time-based that needs precise control. The lesson here isn't "JS is bad" — it's that the line keeps moving toward CSS, and a lot of UI that used to need JS doesn't any more.
