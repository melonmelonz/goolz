Five Free Responses

My answers, in my voice. Each one explains the call, not just the rule.

① const vs let — when do you reach for each?

Both declare a block-scoped binding. The difference is reassignment: const forbids it, let allows it. With const fruits = [] I can still fruits.push(...) all day — const protects the binding, not the contents of the object the binding points at. I default to const because it makes intent obvious at the declaration site: "this name will keep pointing at the same value." I only reach for let when I genuinely need to reassign — loop counters in a classic for, accumulators like let total = 0, a let highest = arr[0] that gets replaced as I walk a list. var I do not use; it has function scope and gets hoisted in ways that surprise me, and there is no situation where it does something const/let cannot.

② What does triple do? What is triple(4)?

function triple(n) { if (n < 0) { return 0; } return n * 3; }

Takes a number n, returns 0 if n is negative, otherwise returns n * 3. So it is "triple, but clamped at zero on the low end." triple(4) returns 12 — 4 is not less than zero, so the guard does not fire and we hit return n * 3. Worth noting: triple(0) returns 0 because 0 < 0 is false, so it falls through to 0 * 3. Both paths happen to give the same answer at zero, which is the kind of accident I would write a comment about if I were keeping this function.

③ Loop fruits and log "1. apple", "2. banana", "3. cherry"

const fruits = ['apple', 'banana', 'cherry']; for (let i = 0; i < fruits.length; i++) { console.log(`${i + 1}. ${fruits[i]}`); }

Used the classic for because the position is part of the output, so I want the index. i + 1 because the human-readable list starts at one but the array starts at zero. If the prefix were not required I would have used for (const fruit of fruits) — cleaner when the index does not matter.

④ Why === over ==? Give a divergent example.

== does type coercion before comparing, which gives results that look correct in casual reading but hide bugs. === requires both type and value to match, so the comparison means what it looks like it means. The classic gotcha: 0 == false is true (both coerce to the same falsy thing), but 0 === false is false because 0 is a number and false is a boolean. The same trap fires on "" == 0, null == undefined, "5" == 5. If I write if (status == 0) and status ever lands as false or "", that branch fires even though I meant "the number zero." === makes me handle that conversion deliberately or not at all. The cost of typing one extra character beats the cost of a silent type-coercion bug at 2am.

⑤ Render people as <li> items: "Alex (Austin)" etc.

const people = [ { name: 'Alex', city: 'Austin' }, { name: 'Sam', city: 'Boston' }, ]; const list = document.getElementById('people-list'); for (const person of people) { const li = document.createElement('li'); li.textContent = `${person.name} (${person.city})`; list.appendChild(li); }

Used for...of because I do not need the index, just each object. textContent not innerHTML — the strings here are mine, but the habit of treating every dynamic string as untrusted means I never have to remember which paths got user input. Same createElement → textContent → appendChild three-line pattern from the books exercise, which is the move I expect to keep using.

⑥ Verbal demo function (peer activity)

Picking the 5-line getHighest from clean.js:

function getHighest(arr) { let best = arr[0]; for (const n of arr) { if (n > best) { best = n; } } return best; }
  • Read literally: function getHighest takes arr, declares best initialized to arr at index zero, walks each n in arr, replaces best when n is greater, returns best.
  • Plain English: walk the array, remember the largest value seen so far, return it.
  • One thing I'd change: best = arr[0] quietly returns undefined if the array is empty, which then propagates through any later comparison. I would either guard at the top (if (arr.length === 0) return null;) or document that the function assumes a non-empty array. Same conversation as the divide-by-zero in getAverage. Boundaries are where bugs live.
