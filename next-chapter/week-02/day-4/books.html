▶ EX3 / 6 — ARRAY OF OBJECTS → DOM

Data In, List Out

An array of {title, author} objects. The page builds the <li>s from the data using createElement and appendChild. The render is wrapped in a function so the same code renders two different lists below.

① Reading list (6 books)

    ② Second call, different data (2 books)

    Same renderBooks(arr, listEl) function, different array, different target <ul>. This is the part that proves the function actually takes its inputs seriously.

      ③ The render function

      function renderBooks(bookArray, listEl) { while (listEl.firstChild) { listEl.removeChild(listEl.firstChild); } for (const book of bookArray) { const li = document.createElement("li"); const titleEl = document.createElement("strong"); titleEl.textContent = book.title; const authorText = document.createTextNode(` -- ${book.author}`); li.appendChild(titleEl); li.appendChild(authorText); listEl.appendChild(li); } }

      ④ Why I built it this way

      • Two child nodes, not innerHTML. None of these strings are user input today, but treating every dynamic string as untrusted means I never have to remember which paths got user input. Habit is cheap.
      • removeChild loop instead of innerHTML = "". Both empty the list. The loop never parses anything, so there is no path for an HTML string to sneak in if I later refactor.
      • Function takes the target list as a parameter. Letting renderBooks reach into the global DOM by ID would couple the function to one specific page. Passing in the <ul> means the same function works for both lists on this page.

      ⑤ Lab notes

      PENN "Data lives in an array, render the array" is the same shape every framework I'll touch later (React, Svelte, Vue) automates. Doing it by hand once means the framework version is no longer magic — it's just the loop I already wrote, with re-renders driven by state changes instead of a refresh.
