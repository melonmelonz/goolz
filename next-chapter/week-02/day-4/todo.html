Render · Mutate · Re-render

The pattern: data lives in tasks, render() reflects it, handlers mutate then call render() again. Same loop I'll hit again in React next phase, hand-rolled here so I see the moving parts.

① Try it

    ② Shape of the data

    Each task is an object so I can carry done state alongside the text. Plain strings would have made the stretch goal (cross-out-on-check) impossible without lookup gymnastics.

    // each task: { text: "thing to do", done: false } const tasks = [];

    ③ The render

    Wipes children, walks tasks, builds an <li> per task. The checkbox toggles task.done directly and adds/removes the done class on its <li> — no full re-render for a cosmetic flip, since the data is already updated.

    function render() { while (list.firstChild) { list.removeChild(list.firstChild); } for (const task of tasks) { const li = document.createElement("li"); if (task.done) li.classList.add("done"); const checkbox = document.createElement("input"); checkbox.type = "checkbox"; checkbox.checked = task.done; checkbox.addEventListener("change", () => { task.done = checkbox.checked; li.classList.toggle("done", task.done); }); const textSpan = document.createElement("span"); textSpan.className = "text"; textSpan.textContent = task.text; li.appendChild(checkbox); li.appendChild(textSpan); list.appendChild(li); } // count line - small UX nicety, also a sanity check while building }

    ④ Add / Clear handlers

    addBtn.addEventListener("click", () => { const newTask = input.value.trim(); if (newTask === "") { setMessage("type something first"); return; } setMessage(""); tasks.push({ text: newTask, done: false }); input.value = ""; input.focus(); render(); }); input.addEventListener("keydown", (event) => { if (event.key === "Enter") addBtn.click(); }); clearBtn.addEventListener("click", () => { if (tasks.length === 0) return; tasks.length = 0; // truncate in place; const forbids reassigning tasks render(); });

    ⑤ Design notes

    • tasks.length = 0 vs tasks = []. const forbids reassigning the binding but allows mutating the array. Truncating in place keeps any other reference to the same array valid. Today nothing else holds a reference, but the habit is correct.
    • Empty input shows a message instead of silently doing nothing. Silent failures are the kind of thing that wastes 30 seconds before you realize the click did nothing. A one-line message is cheaper than confusion.
    • Enter-in-input fires Add. Without it you have to mouse over every time. Small thing, big quality-of-life win — exactly the kind of friction I'd notice as a user immediately.
    • The checkbox toggle does not call render(). The data is already updated; the only DOM work needed is the classList.toggle on that one <li>. Cheaper, and the input focus does not jump.

    ⑥ Lab notes

    PENN "Data lives in an array, render the array, mutate the array, re-render" — this is the same shape as React's useState + JSX, just with render() called manually and the diffing done by reading the DOM blank slate every time. The framework version is faster, but it is literally the loop I just wrote. That demystifies a lot.
