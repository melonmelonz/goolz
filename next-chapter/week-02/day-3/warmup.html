Break D2 on Purpose

Copy greet.html + greet.js first. Then break, save, refresh, observe. One sabotage at a time. The point is not the fix, it's the symptom pattern — what does each kind of bug actually look like in the wild?

① The Setup

cp greet.html greet.bak.html cp greet.js greet.bak.js # now break the live copy. revert from .bak when needed.

② Sabotage 1 — Typo on a getElementById

Change document.getElementById("name-input") to document.getElementById("nam-input"). Save. Submit.

  • Console pops red: Uncaught TypeError: Cannot read properties of null (reading 'value').
  • The element lookup returned null because no element has that id. The next line tries to read .value on null, which crashes.
  • Pattern to remember: "Cannot read properties of null" almost always means a query selector / getElementById missed. Check the id spelling against the HTML.

③ Sabotage 2 — Comment Out preventDefault

Restore. Now comment out event.preventDefault();. Submit.

  • The greeting flashes for a frame, then the page reloads and the greeting is gone.
  • The URL gains ?name=penn. That's the browser's default form behavior: GET against the current page with the inputs as query params.
  • No error in the console. The listener fired correctly. The browser then did its own thing on top, and the navigation wiped the DOM.
  • Pattern to remember: "I see my output for half a second then it vanishes" almost always means an unprevented default. Forms reload. Anchor tags navigate. preventDefault is the off switch.

④ Sabotage 3 — .values Instead of .value

Restore. Now change nameInput.value to nameInput.values. Submit.

  • The greeting renders as Hello, undefined!. No error.
  • .values is not a property of an input element. Reading a missing property gives undefined back, no crash. Template literals happily stringify that as the literal word undefined.
  • Pattern to remember: if your output literally contains the word undefined, you read a property that doesn't exist. JavaScript will not warn you. The console is silent. console.log the variable to confirm.

⑤ Sabotage 4 — Move the Script into <head>

Restore. Move <script src="greet.js"></script> from the bottom of <body> up into <head>. No defer. Save, refresh, click Submit.

  • Console: Uncaught TypeError: Cannot read properties of null (reading 'addEventListener').
  • The script ran before the DOM was parsed. getElementById('greet-form') returned null because at that moment, the form didn't exist yet.
  • The fix is either: keep the script at the end of <body> (what we did originally), add defer to the script tag, or wrap the code in a DOMContentLoaded listener.
  • Pattern to remember: same null-property error as Sabotage 1, but the cause is timing, not spelling. If the id is correct and the element is in the HTML, it's a load-order problem.

⑥ Lab Notes

PENN Two of these four sabotages produced the exact same error string. One was a typo, one was script-order. That's the lesson I actually want: an error message tells you what went wrong, not why. The why is on me, with console.log, the Elements panel, and a slow read.
