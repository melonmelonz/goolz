DevTools Hunt

Open any HTML page I've built this week. Press F12 (or Cmd+Opt+I on Mac). The dev tools panel is the difference between guessing and seeing. Five challenges, one per panel.

① Challenge 1 — Console Tab

  • 2 + 2 → 4. The console is a live JavaScript REPL on the current page.
  • document.title → returns the string in <title> for whatever tab I'm on.
  • document.title = "I changed the title!" → the browser tab text updates immediately. The console can mutate live page state.
  • Lesson: the Console is a sandbox for experiments. If I'm not sure what a function returns, ask it directly here before I write it into a file.

② Challenge 2 — Elements Tab

  • Click the main heading. The Styles panel on the right shows every CSS rule touching it, in cascade order — newest, most specific on top.
  • Double-click a color value, change to red. The page repaints instantly. Nothing was saved to disk; the changes live in the browser memory only.
  • Add a new declaration: type text-decoration: underline; in the element.style block. It applies. This is how I prototype CSS tweaks before committing them.
  • Lesson: Elements + Styles is for "what's actually being applied?" When the page doesn't look right and the CSS file seems correct, the cascade view tells me which rule is winning.

③ Challenge 3 — Box Model

Click any element, switch to the Computed tab. Scroll to the box model diagram at the bottom. Four nested rectangles: content, padding, border, margin.

  • Each side has its own pixel value, not the shorthand. If I wrote padding: 12px;, the diagram still shows 12 / 12 / 12 / 12.
  • Hover over the page element while this is open — the same colored overlay highlights the actual content/padding/border/margin regions on screen.
  • Lesson: the box model is where "why is there extra space here?" questions get answered. It's almost always margin or padding I forgot about.

④ Challenge 4 — Network Tab

Switch to Network, then refresh the page. Every request the browser made appears as a row.

  • HTML, CSS, JS, images, fonts, API calls — all there with status code, size, time.
  • A 404 shows up in red. That's how I catch broken image paths or missing CSS files faster than reading my code.
  • Click any row. The right pane has Headers, Response, Preview. For a CSS file, Preview shows my actual stylesheet. For a JSON API call, Preview gives a tree view.
  • Lesson: if the page is "missing something" and the source code looks fine, Network tells me what the browser actually fetched. Fetched is truth.

⑤ Challenge 5 — Live Editing

  • Back to Elements. Find a <p> on the page. Double-click the text content in the panel. Retype it: "the chicken merchant has a knife."
  • The page updates. Take a screenshot. Refresh the page. The change is gone.
  • Why it disappears: DevTools mutated the in-memory DOM only. The source HTML file on disk is untouched. Refresh re-fetches the original file and starts over. Useful for testing copy changes; useless for persistence.
  • Lesson: DevTools is a sandbox, not an editor. To save a change, I have to put it in the source file. Same goes for CSS edits in Styles — they evaporate on refresh.

⑥ The Hunt Map

When I'm looking for... Open this
JS errors and console.log output Console
"why is this CSS rule not winning?" Elements → Styles
"why is there a gap I didn't ask for?" Elements → Computed → box model
"why isn't this image / API / file loading?" Network
"what does this code return right now?" Console (REPL)
PENN DevTools is the cheat sheet that comes with every browser, and most beginners never open it. After today I never close it.
