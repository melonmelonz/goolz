③ The JS — debug-script.js

function addTask() {
  const input    = document.getElementById("task-input");
  const taskText = input.value;

  if (taskText === "") {
    alert("Please enter a task!");
    return;
  }

  const taskList = document.getElementById("task-list");
  const li       = document.createElement("li");
  li.textContent = taskText;

  const deleteBtn       = document.createElement("button");
  deleteBtn.textContent = "Delete";
  deleteBtn.onclick     = function () {
    taskList.removeChild(li);
  };

  li.appendChild(deleteBtn);
  taskList.appendChild(li);

  input.value = "";
}

④ How I'd Approach It

  1. Open debug-challenge.html, open DevTools, refresh.
  2. Try to add a task. Whatever's wrong shows up either as a console error, a Network 404, or a visual misbehavior.
  3. Triage by panel: Console first (errors), then Elements (broken DOM tree), then Computed (CSS not applying).

⑤ Bug 1 — In the HTML

Look at the list:

<ul id="task-list">
</ol>

Opens with <ul>, closes with </ol>. Different tag. The browser tries to fix it up but the DOM tree ends up unpredictable, and the id="task-list" element doesn't always end up where the script expects.

How I find it: open the Elements tab, look at the rendered DOM tree. The structure won't match the source. Or, if I'm reading carefully, the closing tag jumps out on a slow visual scan.

Fix: close with </ul>.

⑥ Bug 2 — In the CSS

Last rule:

alignitems: center;

That's not a CSS property. The real one is align-items with a hyphen. Browsers silently ignore unknown CSS properties. No console error, no visual hint beyond the items not being vertically centered in the row.

How I find it: open Elements, click an <li>, look at the Styles panel on the right. alignitems appears with a strikethrough or a yellow warning triangle — DevTools flags unknown properties.

Fix: align-items: center;.

⑦ The Clean File

debug-script.js is fine. It builds an li, attaches a delete handler, appends, clears the input. Nothing wrong with it.

Worth saying out loud: I read the JavaScript first because the bug "felt like JS." It wasn't. Once I confirmed the script's logic, I moved on. Confirming a clean file is part of debugging — the alternative is rewriting code that already worked, which is how new bugs get added.
