Greet Me

First real event listener. Type a name, hit submit, watch the DOM actually change. Then poke at it until something breaks.

① The Prompt

Give me an HTML page with a single text input and a button. When the button is clicked, take whatever the user typed and put "Hello, [name]!" into a paragraph on the page. Use addEventListener. Use event.preventDefault() so the form doesn't reload the page. Vanilla JS only.

② What GPT Sent Back

<form id="greet-form">
  <label for="name">Your name</label>
  <input id="name" type="text" required>
  <button type="submit">Greet</button>
</form>
<p id="greeting"></p>

<script>
  const form     = document.getElementById('greet-form');
  const input    = document.getElementById('name');
  const greeting = document.getElementById('greeting');

  form.addEventListener('submit', (event) => {
    event.preventDefault();
    greeting.textContent = `Hello, ${input.value}!`;
  });
</script>

④ Lab Investigations

The brief said: poke it. Three experiments, three things learned.

A. Comment out event.preventDefault()

  • Submit the form → page flashes and reloads. Browser appends ?name=penn to the URL because the form has no action and defaults to GET on the current page.
  • The greeting paragraph briefly shows "Hello, penn!" then vanishes with the reload — the listener fired, then the navigation wiped the DOM.
  • Lesson: the browser's default behavior for a <form> submit is to navigate. If you want to handle it client-side, you HAVE to stop it.

B. Swap textContent for innerHTML

  • Type penn → identical output. Same string, two paths into the DOM.
  • Type <em>penn</em> →
    • with .textContent: shows the literal angle brackets, <em>penn</em>. Treated as text.
    • with .innerHTML: shows penn in italics. The string was parsed as HTML and inserted as a real DOM node.
  • Lesson: .textContent = "this is text, render it as text." .innerHTML = "parse this and build DOM nodes from it." Two very different trust levels.

C. The XSS test

With .innerHTML still wired up, type:

<img src=x onerror="alert('pwned')">
  • The image has no valid src, so it errors. The onerror handler fires. alert() pops.
  • I just executed JavaScript that came from a form input. If the "name" came from a database, a URL parameter, another user's profile, an API — I'd have shipped a stored XSS hole.
  • Lesson: never put untrusted strings through .innerHTML. .textContent is the safe default. Reach for .innerHTML only with content you 100% control, or after sanitizing.

⑤ Final Code (Decision: ship the safe version)

const form     = document.getElementById('greet-form');
const input    = document.getElementById('greet-name');
const greeting = document.getElementById('greeting');

form.addEventListener('submit', (event) => {
  event.preventDefault();                 // 1. don't reload
  const name = input.value.trim();        // 2. read value
  greeting.textContent = `Hello, ${name}!`; // 3. write SAFELY
});
