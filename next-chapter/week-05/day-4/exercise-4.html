Environment Variables

Exercise 4 / 5 . Level 2 . 20 min

Smallest exercise of the week. Probably the most important. Move the API base URL out of the source, into a .env file, read it through import.meta.env.VITE_*, add .env to .gitignore, restart the dev server, verify git status shows no .env. Five steps, two minutes.

The .env file

# .env (project root)
VITE_API_BASE_URL=https://api.github.com

The code change

// before
const response = await fetch(
  `https://api.github.com/users/${encodeURIComponent(username)}`
);

// after
const API_BASE = import.meta.env.VITE_API_BASE_URL;
const response = await fetch(
  `${API_BASE}/users/${encodeURIComponent(username)}`
);

The .gitignore entry

# in .gitignore
.env
.env.local
Why does the variable name need the VITE_ prefix?

Vite's safety rail. By default Vite refuses to expose any env var to client-side code, because anything bundled into the browser is, in effect, public. The VITE_ prefix is an opt-in: "yes, I know this will be readable by anyone who downloads the JS bundle." For things that shouldn't be in the browser (a real secret key for a paid API, anything that should only be hit server-side), you would omit the prefix and access it from a server function, not the React component. The prefix is a hint to me about what I am about to leak.

If VITE_ vars are still public, what was the point?

Two things. Even public-but-configurable beats hardcoded. The same code can hit prod and dev base URLs without editing source. And it is the same mechanism for things that are secrets when you do it on the server side (Cloudflare Worker bindings, Node API routes, etc.). Same syntax, different runtime safety. Learning the habit on the public side means I don't fumble the syntax when it is the real secret on the server side later.

Why restart the dev server after editing .env?

Vite reads env vars at startup, not on every request. Edit the file, the running process doesn't know. npm run dev again, the file gets re-read, the new values land. Forget this and you'll be debugging "the new URL isn't taking effect" for ten minutes wondering why your code looks right but the network tab still shows the old endpoint.

What if you accidentally commit .env before adding it to .gitignore?

Once it is in git history it is effectively public, even if you delete it in the next commit. The secret is in the history, and the history is on GitHub. The fix is to rotate the secret (generate a new key, invalidate the old one) AND scrub the file from history with git filter-repo or BFG. The lesson is: .gitignore first, then create .env. Reversing that order has cost people real money.

What I verified

$ git status
On branch main
nothing to commit, working tree clean

The .env file is present on disk (next to package.json), the app reads it, but git status doesn't see it. That is the configuration working. If I had skipped the .gitignore step, .env would have shown up under "Untracked files", which is the signal to stop and fix the gitignore before doing anything else.

Where this maps to my other work

The Free Dictionary API I used in W4D2 didn't need a key, so I never had to think about this. The GitHub one I am using today doesn't either (rate-limited but not keyed). But the OpenWeatherMap example in the day brief does, and so will most of the third-party APIs I will touch in Phase 2. Getting the hygiene right on a no-key URL today means the muscle memory is already there when the key is real next week.

My PEEK / BeaglePlay work has the analog of this. Never commit ~/.ssh/id_ed25519. Never paste a serial key into a config file that is in source control. The web side is the same shape, just with bots scanning the GitHub firehose instead of a person reading your repo.
