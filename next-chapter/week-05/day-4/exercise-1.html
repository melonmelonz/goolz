Fetch and Display Posts

Exercise 1 / 5 . Level 1 . 30 min

The canonical pattern. Three pieces of state, one useEffect, three render branches. Built it from the starter, traced through what the user sees at each stage, then modified -- post number prefix, rounded card corners, character count under each body. Modifications were trivial because the data plumbing was already right; everything in JSX was just reading from the same posts array.

The component

import { useState, useEffect } from "react";

function PostList() {
  const [posts, setPosts] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    async function fetchPosts() {
      try {
        const response = await fetch(
          "https://jsonplaceholder.typicode.com/posts?_limit=10"
        );
        if (!response.ok) {
          throw new Error("Failed to fetch posts");
        }
        const data = await response.json();
        setPosts(data);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    }
    fetchPosts();
  }, []);

  if (loading) return <p>Loading posts...</p>;
  if (error)   return <p>Error: {error}</p>;

  return (
    <div>
      <h1>Posts</h1>
      {posts.map((post, i) => (
        <article key={post.id} className="post">
          <h2>#{i + 1}: {post.title}</h2>
          <p>{post.body}</p>
          <small>{post.body.length} chars</small>
        </article>
      ))}
    </div>
  );
}
Why is the async function defined inside useEffect instead of being useEffect's callback?

useEffect's callback cannot be async, because React expects it to return either nothing or a cleanup function -- and an async function always returns a Promise, which React would misread as a cleanup. The workaround is what's above: define an async function inside the effect, then call it. Same behavior, no Promise leakage to React. (You can also do (async () => { ... })() as an IIFE, but the named function is easier to read.)

Why the empty array [] at the end of useEffect?

The dependency array tells React when to re-run the effect. [] means "never re-run -- only fire on mount." If I left it off entirely, the effect would run on every render -- and since the effect updates state and state updates trigger re-renders, that's an infinite fetch loop. Pass an empty array on purpose; React lints for missing deps but won't catch missing brackets the same way.

Why response.ok and not just check for data?

fetch only rejects on network failure, not on HTTP error responses. A 500 from the server returns a Response object, not a throw -- await response.json() would happily parse whatever the server sent (maybe an HTML error page) and you'd get garbage in posts. response.ok is true for 200-299 and false for everything else. Throwing on !response.ok is what makes the failure path actually go through the catch.

Trace I wrote out

  1. Component first renders. loading: true, posts: [], error: null. Falls into the first if. User sees "Loading posts...".
  2. After that render commits, useEffect fires. fetchPosts() starts, returns a Promise immediately, function exits. UI is still showing the loading state.
  3. Network round-trip happens (some milliseconds, some seconds). Component is idle.
  4. Response arrives. response.ok is true, await response.json() parses the body, setPosts(data) queues the array. finally runs, setLoading(false) queues the flag.
  5. React batches both updates into one re-render. loading: false, posts: [10 items], error: null. Both early returns are skipped, falls into the JSX return. User sees the list.

What I noticed

The hidden second await -- response.json() -- is the one that trips people up. fetch resolves as soon as headers are available, not when the full body has arrived. response.json() reads the body and parses it, which is its own async op. The day brief's "envelope vs letter" framing is correct and I want to keep it. Skip the second await and you get a Promise where you wanted an object, and the JSX renders [object Promise], which is exactly the kind of error that makes you stare for thirty seconds.

Modifications were small. Post number is just i + 1 -- the second arg to .map is the index. Character count is post.body.length. Rounded corners is CSS. None of it needed any state changes, because all three are just functions of data that's already there. That's the payoff of the pattern: once the data is in state, you can read any derived value from it for free.
