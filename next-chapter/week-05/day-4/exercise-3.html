GitHub User Search

Exercise 3 / 5 . Level 2 . 45 min

Lookup tool. Type a username, click search, render the profile. The shape that's different from Ex 1: the fetch fires on click, not on mount. So useEffect doesn't get involved at all -- the async function runs straight out of the submit handler. Same three states (loading/data/error), same try/catch/finally, just kicked off by a user action instead of a mount.

The component

import { useState } from "react";

function GitHubSearch() {
  const [username, setUsername] = useState("");
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  async function handleSearch(e) {
    e.preventDefault();
    if (username.trim() === "") return;

    setLoading(true);
    setError(null);
    setUser(null);

    try {
      const response = await fetch(
        `https://api.github.com/users/${encodeURIComponent(username)}`
      );
      if (response.status === 404) {
        throw new Error("User not found");
      }
      if (!response.ok) {
        throw new Error(`GitHub returned ${response.status}`);
      }
      const data = await response.json();
      setUser(data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }

  return (
    <div>
      <h1>GitHub User Search</h1>
      <form onSubmit={handleSearch}>
        <input
          value={username}
          onChange={(e) => setUsername(e.target.value)}
          placeholder="github handle"
        />
        <button type="submit" disabled={loading}>Search</button>
      </form>

      {loading && <p>Searching...</p>}
      {error   && <p className="err">{error}</p>}

      {user && (
        <article className="profile">
          <img src={user.avatar_url} alt="" width={96} height={96} />
          <h2>{user.name || user.login}</h2>
          <p>{user.bio || "No bio available"}</p>
          <p>Public repos: {user.public_repos}</p>
          {user.location && <p>Location: {user.location}</p>}
          <a href={user.html_url} target="_blank" rel="noopener noreferrer">
            View on GitHub
          </a>
        </article>
      )}
    </div>
  );
}
Why no useEffect here?

Because the fetch is event-driven, not mount-driven. useEffect is the right tool when "when this thing first appears on screen, run X." When the trigger is a click instead, just call the async function from the click handler -- the handler itself is the equivalent of the effect. Reaching for useEffect on click-driven fetches is a common over-application of the hook. It's not needed and it actively makes the code harder to follow.

Why reset user/error/loading at the top of handleSearch?

Without it, a second search runs with the first search's result still in state. The flicker would be: previous user still on screen, loading flag goes up, new user arrives, swap. With the reset: previous user disappears immediately, loading state shows, new user arrives. Less confusing. Same lesson as making sure interim states are explicit -- the user should never see stale data labelled as if it's fresh.

encodeURIComponent on username -- why?

Defensive. Most GitHub handles are URL-safe (alphanumerics and hyphens) but the input is user-controlled. If somebody types a space or a slash, the URL breaks in a way that's annoying to debug. encodeURIComponent escapes anything that would break the URL. It's the same one-line habit as parameterized SQL queries -- always do it, never trust the input shape, the cost is nothing.

disabled={loading} on the button -- why?

So the user can't fire a second request while the first is in flight. Without it, a frustrated user clicking three times fires three requests, three setStates, and the responses can arrive out of order (race condition -- last-write-wins on state, not last-fired). Disabling the button while loading is the cheap fix. The proper fix is request cancellation with AbortController, which is a Week 6 topic the day brief flagged. For now: disabled button, one request at a time.

What I noticed running it

Searched my handle, got my profile. Searched octocat, got the GitHub mascot. Searched thisuserdefinitelydoesnotexist1234, got "User not found" -- the 404 path. Watched the network tab: clean 200, clean 404, no console errors. The on-submit pattern handles all three.

The rate-limit warning in the day brief is real. GitHub's unauthenticated limit is 60 requests per hour per IP. I burned through six during testing without thinking; that's 10% gone in the first minute. The day brief's advice -- only fetch on Search click, never on every keystroke -- is the only thing that keeps you under the limit. Same energy as not blasting your AWS API key every render.

What I'd build on top

Search the user's repos as a second call (https://api.github.com/users/{username}/repos) and render the top three by star count. That's a second fetch chained off the first -- which raises the question of whether to fire both in parallel or sequentially. Parallel with Promise.all is the right move when neither depends on the other, which is the case here (you can fetch the user and the repos at the same time off the username). One more pattern I want to land before the week is out.
