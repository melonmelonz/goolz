Day 4 -- Putting It Together: React + APIs

NCD . WK5 . D4 . five exercises loaded . the day the data shows up from somewhere else

Today is the day the data stops being hardcoded. Up to now, every array in every component has been a const sitting at the top of the file. Today the array gets fetched from a URL and arrives whenever it arrives -- which means everything I just learned about useState and map and conditional rendering has to also handle "the data isn't here yet" and "the data tried to come and something went wrong." Three states for one render path. That's the pattern of the whole day.

The async stuff isn't new to me from yesterday -- I've been pumping bytes around in Rust and through SPI buses for the last three weeks, where everything is "fire a transaction, wait for it to complete, deal with whatever it returns." The DMA-driven SPI transfers I'm wrangling on the BeaglePlay are exactly the same shape as fetch: kick off an op, get a future back, await it. JavaScript's Promise is just a userland version of the same thing. What's actually new today is the React-shaped wrapper around it -- useEffect, the three-state pattern, the cleanup-on-unmount fence.

Morning Warm-up -- What Order?

Predict: A, C, B.

Why: JS doesn't block on setTimeout. It queues the callback on the event loop and moves on. console.log("C") runs synchronously right after setTimeout is called (not after the timer fires). One second later the event loop drains the queued callback and "B" prints. This is the exact problem the rest of the day solves -- async behavior is the default, and async/await is the syntax that lets you write it in linear-looking code.

Verbal Mini-Demo -- Loading / Data / Error

Pair partner asked me to walk the three-state pattern. Pulled up the JSONPlaceholder posts from Ex 1: posts starts as [], loading starts as true, error starts as null. On mount, useEffect fires once (empty deps), kicks off the async fetch. While the request is in flight the component renders "Loading posts..." -- which is the only branch the user sees right now because loading is true.

When the promise resolves: setPosts(data) stages new data, setLoading(false) stages the flag flip, the finally block guarantees that flag flip happens whether the fetch succeeded or failed. React batches the two state updates into one re-render, the new render evaluates the conditionals fresh -- loading is now false, error is still null, falls through to the data branch, renders the list.

The error branch is the same shape, just driven by catch. The trick I want to remember: finally is what makes the pattern symmetric. Without it, the failure branch would forget to flip loading and the spinner would spin forever on errors. Try/catch and finally is the trio that keeps the state machine honest.

The Thing That Clicked Today

Data fetching in React isn't a feature -- it's a discipline that React makes uncomfortable on purpose. useEffect with an empty dep array is the framework forcing you to be explicit about when the side effect runs ("once, when this thing first appears"). The three states aren't React's fault either -- they're the fault of the network being slow and unreliable. React is just refusing to pretend that data is synchronous, which is the same honesty I want out of any system: don't hide the cost, show me the cost so I can decide what to do about it.

The env-var stuff in Ex 4 is the smallest exercise but it's the one I needed most. I've been sloppy about secrets in personal projects, and the "bots find your API key within minutes" line in the day brief is not hyperbole -- I've watched it happen to someone in the cohort during week 4. Adding .env to .gitignore and reading import.meta.env.VITE_* instead of hardcoding URLs is a thirty-second habit that saves a hundred-dollar surprise later. Starting it on every project from today forward, no exceptions.
