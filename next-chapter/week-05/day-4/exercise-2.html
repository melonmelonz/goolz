Error Handling

Exercise 2 / 5 . Level 1 . 20 min

Intentionally break the URL. Verify the error state actually fires. Replace the bare error message with something a real user could read. Then fix the URL and confirm nothing else regressed. Small exercise, real habit.

Breaking it

Changed posts to BROKEN in the URL. Saved, reloaded, watched: "Loading posts..." for a beat, then "Error: Failed to fetch posts." Good -- the !response.ok branch threw, the catch caught it, setError populated, finally flipped loading off. The error path is wired the same as the success path -- both run through finally, so the spinner doesn't get stuck. That symmetry is what makes the try/catch/finally pattern actually correct, not just defensive.

The user-friendly fallback

if (error) {
  return (
    <div className="err-card">
      <h2>Something went wrong</h2>
      <p>We couldn't load the posts. Check your connection and try again.</p>
      <p className="err-detail"><code>{error}</code></p>
      <button onClick={() => window.location.reload()}>Retry</button>
    </div>
  );
}
Why show the raw error text at all?

For me, not for the user. The big copy ("We couldn't load the posts...") is what a normal user reads and acts on. The small <code> line below is a hook for triage -- if someone screenshots this and sends it to me, I want to be able to tell "404" from "network offline" from "CORS blocked" without asking. It's the same instinct as a kernel panic that prints both a friendly message and the stack trace: humans get one, debuggers get the other.

Why window.location.reload() instead of a proper retry?

Cheapest possible thing that works. A real retry would re-call the fetch function from inside React state, without reloading the page -- which means lifting the fetch into a callback I can re-invoke, and resetting loading/error/data back to their initial values. That's a refactor I'd do in a real app. For a lab error state it's fine to nuke and reload, because the cost of doing it right is bigger than the cost of the workaround.

What about a network failure (offline)?

Different code path. response.ok only matters if the request actually completed. If the browser is offline, fetch rejects with a TypeError -- "Failed to fetch" or similar. That throw lands in the same catch block and produces the same error state. The error message is different (the user sees a different err.message in the small line), but the UX is the same. The pattern handles both with one branch, which is exactly the point.

Fixed and verified

Put posts back in the URL. Reloaded. Loading state for a beat, then the list rendered. Diffed the file against pre-break to make sure nothing else regressed -- only the URL string changed (and the error UI was kept, because there's no reason to remove the friendly fallback once you have it).

What I noticed

Every fetch I write from now on gets the friendly-error UI from the start, not as a "do this later." It's the kind of thing that's free to add at the start (you're already writing the error branch, you might as well make it useful) and expensive to retrofit later (you have to find every fetch in the codebase). Habit, not feature.

The smaller observation: the day brief is right that users leave on a white screen. The cost of a 30-second friendly-error component is approximately zero. The cost of losing a user because they hit a flaky API call is not zero. The asymmetry is the argument.
