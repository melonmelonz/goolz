Day 2 -- React: Thinking in Components

NCD . WK5 . D2 . five exercises loaded

Yesterday Node taught me that JavaScript can leave the browser. Today React teaches me how to put it back in a way that scales past a single HTML file. Components instead of pages. The interesting part is that I already think this way for everything else I write -- Rust crates have modules, Python files have functions, the kernel has subsystems. Front-end was the holdout. Today's the day it catches up.

The mental model the day brief landed on -- UI = f(state) -- is the thing I want to actually feel by the end of lab. Not just nod at. The "you describe what it should look like and React figures out the diff" pitch is exactly how I want to write UI code, because the alternative (manual DOM mutation) is what made me dread anything that needed JavaScript in Phase 1.

Morning Warm-up -- Spot the Components

Q1 -- how many distinct component types? Four. NavBar, Card, Footer, and the outer Page that holds them. The four cards look identical but they're the same type with different data; only the type counts.

Q2 -- which piece first? The Card. It's the smallest, most-repeated unit and everything else is just chrome around it. Build the leaf, then frame it.

Q3 -- what data per card? An image URL, a title, a description. Probably also an id once you start mapping over a list. That's exactly the props signature: { image, title, description }.

Verbal Mini-Demo -- Props vs State

Pair partner asked me to do the props vs state walk. I pulled up the favorites list from Ex 5 and pointed: FavoriteItem takes text and onRemove as props -- it never changes them, it just reads them and renders. The parent owns the truth. App owns favorites and inputValue as state -- those are the things that change in response to clicks and typing, and changing them is what causes the screen to update.

The rule I tried to land: if a value is given to a component, it's a prop. If a value belongs to the component and the component is allowed to change it, it's state. Same data can be state in one component and a prop in its child. That's what "lifting state up" is going to mean in a later week, but the seed is here.

Their follow-up: "what actually happens on a re-render?" Honest answer: React calls the function again with the new state, builds a fresh virtual DOM tree, diffs it against the previous tree, and only patches the real DOM nodes that changed. The function runs from the top every time -- which is why useState looks weird (how does it remember?) but actually works (React keeps the state slot, your function just reads from it).

The Thing That Clicked Today

UI = f(state). Not the equation -- the inversion of control. In Phase 1 I was writing code that said "find this element, change its text." Today I wrote code that says "here is what the screen should look like given this data," and React figures out the find-and-change step. That's the same shift I felt going from C-style mutation to Rust's match on a fresh value: stop telling the runtime what to do, start describing what you want the result to be.

The other thing is how cheap composition gets. The favorites app is a 90-line file but it's actually three components with clean seams between them. I could yank FavoriteItem into a separate file and import it tomorrow without changing one line of App. That's not a feature of React -- it's a feature of having a real module system layered onto a real component model. Yesterday's modules + today's components is the actual point. Each was useful alone. Together they're the whole game.
