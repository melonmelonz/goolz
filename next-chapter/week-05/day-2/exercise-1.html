Read a React Component

Exercise 1 / 5 . Level 1 . 25 min

Predict-before-run. The point of the exercise is to confirm I can read JSX without executing it, the same way I should be able to read a function signature and trace it on paper. I read the component first, answered the questions, then spun up Vite. The predictions were all right except for one off-by-one I caught reading my own answer back.

The component

function MovieCard({ title, year, rating, genre }) {
  return (
    <div className="movie-card">
      <h2>{title} ({year})</h2>
      <p>Genre: {genre}</p>
      <p>Rating: {"*".repeat(rating)}</p>
      {rating >= 4 && <p className="recommended">Highly Recommended!</p>}
    </div>
  );
}
What props does MovieCard accept?

Four: title (string), year (number), rating (number, looks like 1-5), genre (string). They're destructured out of the props object in the function signature, which is the same pattern as destructuring a struct in Rust or a dict in Python -- pull the named fields out, ignore the rest.

For "The Matrix", what will the <h2> display?

The Matrix (1999). JSX evaluates the {title} and {year} expressions and stitches them into the text with the literal space and parens around them.

What does "*".repeat(rating) do for rating 5 vs rating 2?

5 stars: "*****". 2 stars: "**". String.prototype.repeat(n) returns a string with the original repeated n times. Same primitive Python has with "*" * n. Quick-and-dirty star bar, no SVG needed.

{rating >= 4 && <p>...</p>} -- what does this do?

JS short-circuit rendering. If rating >= 4 is false, the expression evaluates to false and React renders nothing. If it's true, the expression evaluates to the <p> element and React renders it. The Matrix (5) and Toy Story (4) get the "Highly Recommended!" line; A Bad Movie (2) doesn't. This is the React idiom for conditional rendering -- no ternary needed when the alternative is "nothing."

Adding a director prop -- two places to change?

One: add director to the destructured parameter list. Two: render it somewhere in the JSX, e.g. <p>Directed by {director}</p>. And then a third place if you're being honest: every <MovieCard> call site needs the new prop or it'll render undefined. Not a hard error, just a quiet one -- which is exactly the kind of thing TypeScript would catch and plain JSX won't.

Then I ran it

Set up movie-app with npm create vite@latest movie-app -- --template react, pasted the App.jsx, hit npm run dev, opened localhost:5173. Three cards. Matrix and Toy Story both flagged "Highly Recommended!" in green-bold (the inline style version). A Bad Movie quietly didn't. Predictions confirmed.

One thing I noticed running it that I didn't notice reading it: the recommended <p> sits inside the same border as the rest of the card, no spacing tweak needed. That's because it's just another child of the card <div>. No magic. The "conditional" part is in the data, not in the DOM structure.

Modification

Added a fourth card -- The Lighthouse (2019, 5 stars, "Horror") -- and a director prop on the component. Final App.jsx looked like this for the relevant chunk: 

function MovieCard({ title, year, rating, genre, director }) {
  return (
    <div style={{ /* ...same... */ }}>
      <h2>{title} ({year})</h2>
      <p>Genre: {genre}</p>
      <p>Director: {director}</p>
      <p>Rating: {"*".repeat(rating)}</p>
      {rating >= 4 && <p style={{ color: "green", fontWeight: "bold" }}>Highly Recommended!</p>}
    </div>
  );
}

<MovieCard
  title="The Lighthouse"
  year={2019}
  rating={5}
  genre="Horror"
  director="Robert Eggers"
/>

Existing cards (Matrix, Toy Story, A Bad Movie) now render Director: undefined because I didn't backfill them. Caught it on the first render, fixed it by adding the director on each call site. Real lesson: changing a prop's interface is a refactor across every call site. That's not a React thing; that's any function. The friction is the same.

What I noticed

Reflection

The destructuring-in-the-signature style (function MovieCard({ title, year, ... })) is much nicer to read than function MovieCard(props) followed by props.title everywhere. It's the same idea as named arguments in Python. The "what does this component take?" question is answered in the signature, not buried in the body.

The short-circuit {cond && <jsx />} trick is the kind of thing that looks clever the first time and then becomes second nature. The one footgun the day brief didn't flag: if cond is the number 0, JSX will render the literal 0 on screen instead of nothing, because 0 && anything is 0, not false. Easy to hit when you're conditioning on array length. Filing that one for later.
