Compose a Page

Exercise 4 / 5 . Level 2 . 30 min

Composition. The exercise where the whole point of components stops being theoretical. App renders Header, Main, and Footer. Main renders three ProfileCards. Four components total, one screen, and each piece can be changed without breaking the others.

The structure

App
  ├─ Header (props: appName, subtitle)
  ├─ Main
  │   └─ ProfileCard × 3 (props: name, role, bio)
  ├─ Contact (props: email, message)  ← my addition
  └─ Footer

This is the tree I built. The relevant insight is that Main doesn't know App exists, and App doesn't know what Main renders inside itself. Each component is a black box from the outside. From App's view, Main is just "this opaque thing I drop into the page." That's the seam.

Header takes a subtitle now

function Header({ appName, subtitle }) {
  return (
    <header style={{
      backgroundColor: "#1a1a2e",
      color: "white",
      padding: "20px",
      textAlign: "center"
    }}>
      <h1 style={{ margin: 0 }}>{appName}</h1>
      {subtitle && (
        <p style={{ margin: "8px 0 0 0", color: "#aaa" }}>{subtitle}</p>
      )}
    </header>
  );
}

<Header appName="PEEK" subtitle="examines embedded kernels" />

I picked PEEK as the app name because that's what I'd actually build a page like this for -- the capstone needs a landing page eventually and "Header / Main / Footer" is the minimum useful shell. The {subtitle && ...} short-circuit means I can drop the subtitle prop on a page that doesn't want one and the header just collapses. Optional props with sensible defaults, no destructuring tricks needed.

Contact section added between Main and Footer

function Contact({ email, message }) {
  return (
    <section style={{
      backgroundColor: "#f4f1e8",
      padding: "24px",
      textAlign: "center",
      borderTop: "1px solid #ddd",
      borderBottom: "1px solid #ddd"
    }}>
      <h2>Get in touch</h2>
      <p>{message}</p>
      <p>
        <a href={"mailto:" + email}>{email}</a>
      </p>
    </section>
  );
}

Built as its own component on purpose, even though it's tiny. The day brief asked for it that way and the reason is right: if I inlined the contact markup in App, I'd be back to writing one big file. Putting it in a function with named props makes it a unit I can drop into another page tomorrow.

The whole App, composed

function App() {
  return (
    <div style={{ fontFamily: "Arial, sans-serif", minHeight: "100vh", display: "flex", flexDirection: "column" }}>
      <Header appName="PEEK" subtitle="examines embedded kernels" />
      <Main />
      <Contact
        email="penn@goolz.org"
        message="If you want to hear about progress on the eldritch-tamagotchi TUI, the kernel module, or the BeaglePlay demo -- send a note."
      />
      <Footer />
    </div>
  );
}

Things I noticed

Props flow down. State stays local.

App hands appName and subtitle down to Header. Hands email and message down to Contact. The leaves (Header, Contact) receive data and render. They don't reach up to grab anything. That one-way flow is what makes the tree easy to reason about -- you trace a value by following props from the root outward.

Each component has its own little world.

Footer calls new Date().getFullYear() inside itself. That's a side computation, scoped to the component. App doesn't have to pass the year down. If I want every component to know the year, props is the way. If only one component needs it, just compute it there. The discipline is "lift what you need to share, keep what you don't."

The cost of a new component is low.

Adding Contact took maybe four minutes including the inline styles. I didn't have to register it anywhere, configure routing, edit a manifest. I wrote a function and used it. Compared to the friction of adding a new template partial in some frameworks I won't name, this is the right amount of ceremony: none.

What I learned

Reflection

The "compose small pieces" pattern is the one part of front-end I've been waiting for. Phase 1 HTML was a single page that grew until it was hard to read. Phase 2 modules let me split JavaScript across files. Today is the day the markup gets the same treatment -- I can break the page into named pieces with explicit interfaces, and the page is still just one page from the user's side. Same end result, much better source layout.

The natural next question is "where does data that multiple components need actually live?" Right now everything is local or passed down through props. If Footer wanted to show the current count from ColorCounter, I'd have to hoist that state up to App and pass it down both sides. That's "lifting state up" and it's the next thing the curriculum will hit. I can feel the shape of it from here.
