Favorites List

Exercise 5 / 5 . Level 3 . 35 min

A real little app. Input, button, list, delete. State for the list, state for the input. Two pieces of state, one child component, and a handler per action. This is the smallest interactive React app I'd consider non-trivial. Built it, then did all three modification levels.

I renamed the title to "BeaglePlay TODO" because I'd actually use this for that. Stuff I need to verify on the board between now and 2026-08-10. Real list, fake demo data below.

The state, the handlers, the child

const [favorites, setFavorites] = useState([]);
const [inputValue, setInputValue] = useState("");

const handleAdd = () => {
  if (inputValue.trim() === "") return;
  if (favorites.some(item => item.text === inputValue.trim())) {
    // Level 3 mod: no duplicates
    alert("Already on the list.");
    return;
  }
  const newItem = { id: Date.now(), text: inputValue.trim() };
  setFavorites([...favorites, newItem]);
  setInputValue("");
};

const handleRemove = (idToRemove) => {
  setFavorites(favorites.filter((item) => item.id !== idToRemove));
};

const handleClearAll = () => setFavorites([]);

const handleKeyPress = (event) => {
  if (event.key === "Enter") handleAdd();
};

Code reading -- the bits I want to remember

Two pieces of state -- what each tracks.

favorites is the actual list, an array of { id, text } objects, starts empty. inputValue is whatever's typed in the input box right now, starts as "". They're separate because they change for separate reasons: input changes on every keystroke, the list changes only on Add or Remove. Lumping them together would mean a list re-render on every keystroke, which is wasteful and conceptually wrong.

What props does FavoriteItem receive?

text (the string to display) and onRemove (a function to call when X is clicked). That's the whole interface. FavoriteItem doesn't know anything about the list, the input, or the id. It just renders text and a button. The parent owns the logic; the child owns the rendering. Clean seam.

What happens when Add fires?

1) Trim check -- if the input is empty/whitespace, return without doing anything. 2) Dedup check (my mod) -- if the trimmed text already exists in the list, alert and return. 3) Build a new item with a fresh id (Date.now() as a quick unique key) and the trimmed text. 4) setFavorites([...favorites, newItem]) -- spread the existing array into a new array and append the new item. Crucially, this creates a new array; the old one is left alone. React sees a new reference, re-renders the list. 5) setInputValue("") clears the input.

Why id on each item, not just the index?

The day brief flagged this and it's the right call. React uses the key prop to identify which list item is which between renders. If I used the array index as the key, then deleting item 2 from [A, B, C, D] would leave [A, C, D] -- but React would think item at index 1 just changed text from B to C, item at index 2 changed from C to D, item at index 3 was removed. State attached to those items (focus, animation, internal component state) would get the wrong owners. With stable ids tied to the data itself, React knows "the item with id 1700000001 is gone; the items with ids 1700000002 and 1700000003 are still here, just shifted in the array." Same DOM nodes, just reordered. Date.now() works for this small case because clicks can't happen within the same millisecond from a human. For anything bigger I'd use crypto.randomUUID().

onChange={(event) => setInputValue(event.target.value)} -- the controlled input pattern.

The input's value is driven by React state. Every keystroke fires onChange, which updates inputValue, which re-renders the input with the new value. The DOM input doesn't have its own truth -- React does, and the input just reflects it. That's a "controlled component." It's heavier than letting the DOM hold the value, but it means I can read or modify the input value from anywhere in my component code at any time. Worth the cost for anything past the most trivial form.

Spread operator vs Array.push.

setFavorites([...favorites, newItem]) builds a new array. favorites.push(newItem) mutates the existing one. React relies on identity comparison (oldArray === newArray) to decide whether to re-render the list. push doesn't change the reference; spread does. So push would silently fail to update the UI. This is the kind of footgun I'll definitely hit at least once before it becomes muscle memory.

handleClearAll is one line: setFavorites([]). The button itself is conditionally rendered -- if the list is empty there's nothing to clear, so the button doesn't appear. Subtle styling so it doesn't dominate. The danger of "clear all" without confirmation is real but for a TODO list with no persistence, fine.

Level 3 -- no duplicates

Two ways to do this. I went with the simpler: inside handleAdd, check favorites.some(item => item.text === inputValue.trim()) before adding, and bail with an alert if true. Alert is the lazy choice; the nicer move would be to flash the input border red for a second, but that needs another piece of state for "show error" plus a setTimeout to clear it. Filing that for after I learn useEffect, which is tomorrow's hook.

Final app

Three components -- App, FavoriteItem, and the input is technically just JSX in App. Could extract that into AddForm as a fourth component to be cleaner, but the input + button + handler is small enough that I left it inline. Knowing when to extract is the judgment call. My rule of thumb after one day of React: extract when the same pattern shows up twice, or when a chunk grows past 15 lines of JSX. Both bar would justify pulling out the form. Today, neither does.

Reflection

The whole exercise is one application of UI = f(state). The page is a function of two values, favorites and inputValue. Every interaction is just a handler that produces new values. I never touched document, never selected an element, never set innerHTML. I changed the data, and the screen followed. That is the unlock. I felt it specifically when I added the Clear All button: I wrote setFavorites([]), saved, and the entire list vanished from the page. No "loop through the list and remove each child node." Just data goes away, UI goes away. The DOM step is React's problem.

The thing I want to keep an eye on: as components get bigger, the prop chains get longer. FavoriteItem took two props today. In a real app the leaf components take eight, the parents pass twelve, and threading state through six layers of components ("prop drilling") becomes tedious. I know this is what Context and stores exist to solve, but I want to feel the pain a little first so the solution makes sense when it shows up. For now, two props is fine.
