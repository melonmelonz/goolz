State Playground

Exercise 3 / 5 . Level 2 . 30 min

This is the exercise where the React mental model has to actually click, because before useState a "component" is just a fancy template engine. State is what makes it a thing that can respond to time. Built all three components -- Counter, Toggle, ColorCounter -- in one App.

Counter -- the basics

import { useState } from "react";

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <h2>Counter: {count}</h2>
      <button onClick={() => setCount(count + 1)}>+ Add 1</button>
      <button onClick={() => setCount(count - 1)}>- Subtract 1</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}
What is useState(0) actually doing?

Asking React to allocate a slot of state inside this component, initialized to 0 the first time the component mounts, and returning a 2-tuple: [currentValue, setterFunction]. The destructuring const [count, setCount] = ... just names those two slots. On every subsequent render, useState(0) ignores the initial value and returns the current value of that slot. The "0" only matters on the first call.

Why doesn't setCount(count + 1) mutate count directly?

Because count is a local const inside this render. The setter doesn't change this render's count -- it tells React "next time you render this component, the value for this slot should be count + 1." Then React re-runs the function. The new run sees the new value. That immutability is the whole reason React can do the diff -- old value and new value coexist long enough to compare.

Toggle -- booleans and ternaries

function Toggle() {
  const [isOn, setIsOn] = useState(false);
  return (
    <div>
      <h2>Status: {isOn ? "ON" : "OFF"}</h2>
      <button
        onClick={() => setIsOn(!isOn)}
        style={{ backgroundColor: isOn ? "#2ecc71" : "#e74c3c", color: "white" }}
      >
        {isOn ? "Turn OFF" : "Turn ON"}
      </button>
    </div>
  );
}
New syntax notes.

isOn ? "ON" : "OFF" is a ternary. JS borrowed it from C; same shape in Rust (if isOn { "ON" } else { "OFF" }) just without the question mark. !isOn flips a boolean. setIsOn(!isOn) reads the current value and asks React to set the next one to the opposite. Nothing here is React-specific -- it's just JS expressions in a JSX shell.

Same data, three render slots.

The isOn boolean drives three different bits of output on the same render: the status text, the button color, and the button label. That's the lesson under the lesson -- one piece of state can be projected into many places in the UI, and they all stay in sync automatically because they're all just expressions over the same value.

ColorCounter -- derived values

function ColorCounter() {
  const [count, setCount] = useState(0);

  const getColor = () => {
    if (count >= 10) return "#e74c3c";
    if (count >= 5) return "#f39c12";
    return "#2ecc71";
  };

  return (
    <div>
      <h2 style={{ color: getColor(), fontSize: "48px" }}>{count}</h2>
      <p>
        {count < 5 && "Keep going!"}
        {count >= 5 && count < 10 && "Getting warm..."}
        {count >= 10 && "You made it to 10!"}
      </p>
      <button onClick={() => setCount(count + 1)}>+ 1</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}

Code tracing

count = 3 -- color and message?

Color: #2ecc71 (green -- under 5). Message: "Keep going!" (the first branch is true, the others are false).

count = 7 -- color and message?

Color: #f39c12 (orange -- between 5 and 10). Message: "Getting warm..." (the middle branch fires).

count = 10 -- color and message?

Color: #e74c3c (red -- at or above 10). Message: "You made it to 10!" The check is count >= 10, not >, so 10 itself triggers the red state, not 11. Off-by-one is the kind of thing I want to read carefully every time.

What does Reset do?

Sets count back to 0. Which then re-renders the component: number turns green again, message flips back to "Keep going!". No special "reset" logic -- it's just setCount(0). The UI is downstream of the state.

What clicked here

Reflection

The getColor function is the part I want to remember. It's not state. It's not a prop. It's a plain function defined inside the component body that computes a derived value from state on every render. The temptation in a non-React mindset is to store the color as state and write an effect that updates it when count changes. That would work and be wrong. The color is a pure function of the count. Compute it, don't store it. State is for things that actually change independently; derived data should just be re-derived. That's a discipline-not-syntax lesson and it's the kind of thing I'll get wrong a few times before it sticks.

The "how does useState remember between renders?" question is still open in my head. The honest answer the day brief didn't quite give: React keeps a list of state slots keyed by call order inside the component, and useState reads from the next slot each time. That's why you can't put useState inside a conditional -- the call order would shift and the slots would get crossed. Filing that one as the "rules of hooks" thing I'll probably see explicitly next week.
