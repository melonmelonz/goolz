Build ProfileCard

Exercise 2 / 5 . Level 1 . 30 min

First component I wrote myself today. Stayed close to the day brief's starter, then swapped in real names from the cohort instead of placeholders. The thing I wanted to feel was the moment where the same component renders three times with different data -- which is the entire reason React exists. It hit on the first run, which felt slightly anticlimactic in the way the good ones always do.

Things I picked up doing this

Inline style is a JS object, not a string.

The double-braces in style={{ ... }} aren't React syntax. The outer brace is "I'm putting JS inside JSX," and the inner brace is the JS object literal. Property names are camelCase (borderRadius, not border-radius) and the values are strings. Once you see it as "object literal inside an expression slot" it stops looking strange.

The color prop and string concatenation.

I used border: "2px solid " + color instead of building a template literal, because the prop is just a hex string and JS string concat is fine. Could also have done ` 2px solid ${color}` with template literals -- same result, slightly more idiomatic. Worth noting that the value has to be a string at the end of the day, since that's what CSS wants. The color prop is just data; the conversion to a style string happens at the boundary.

Three cards, one component definition.

This is the part the day brief was selling. The ProfileCard function exists once. The three <ProfileCard /> elements are three calls with different inputs. If I want a fourth card I add a fourth element. If I want to change how every card looks, I edit the function once. That's the whole pitch and it works exactly the way it claims to.

Level 2 modification -- skills as chips

Added a skills prop that takes a comma-separated string and renders each skill as a small chip element. Split, trim, map. 

function ProfileCard({ name, role, bio, color, skills }) {
  const skillList = skills ? skills.split(",").map(s => s.trim()) : [];

  return (
    <div style={{ /* ...same border using color... */ }}>
      <h2>{name}</h2>
      <p style={{ fontStyle: "italic" }}>{role}</p>
      <p>{bio}</p>
      <div style={{ marginTop: "10px", display: "flex", flexWrap: "wrap", gap: "4px" }}>
        {skillList.map((skill) => (
          <span
            key={skill}
            style={{
              fontSize: "11px",
              padding: "2px 8px",
              border: "1px solid " + color,
              borderRadius: "10px",
              color: color
            }}
          >
            {skill}
          </span>
        ))}
      </div>
    </div>
  );
}

<ProfileCard
  name="Penn Porterfield"
  role="Student / Systems Track"
  bio="..."
  color="#5E957A"
  skills="Rust, Linux kernel, SPI, Vim"
/>

The interesting bit is .map((skill) => ( ... )) returning an array of JSX elements that React renders inline. That's the same pattern Ex 5 will use for the favorites list, just one exercise early. The key={skill} is required because React wants a stable identifier per item in an array -- I'm using the skill string itself, which works because skills in a single card shouldn't repeat. If they could repeat, I'd index them or attach an id.

What I learned

Reflection

The mental model that helped: a component is a function that returns markup, and props are its arguments. That's it. There's no special "component machinery" once it's in your head -- it's just a JS function with rules about what its return value looks like. Everything else (state, effects, context) is layered on top of that one primitive.

One thing I'll do differently next time: stop putting inline styles in the JSX. It's fine for one exercise but the styles drown the structure. CSS modules or even a plain className + external sheet would make this readable. Filing that for whenever the course says "now let's talk about styling React."
