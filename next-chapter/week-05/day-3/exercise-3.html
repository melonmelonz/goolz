Search + Filter

Exercise 3 / 7 . Level 2 . 30 min

Add a search box. Type, the list narrows. This is where the data-as-the-truth idea pays its first dividend: the list isn't being mutated -- it's being projected through a filter. The original products array doesn't change. What changes is which subset of it gets handed to .map() on each render.

The change

function ProductList() {
  const [searchTerm, setSearchTerm] = useState("");

  const visible = products.filter((p) =>
    p.name.toLowerCase().includes(searchTerm.toLowerCase())
  );

  return (
    <div>
      <h2>Products</h2>

      <input
        type="text"
        value={searchTerm}
        onChange={(e) => setSearchTerm(e.target.value)}
        placeholder="Search products..."
      />

      {visible.map((p) => (
        <ProductCard
          key={p.id}
          name={p.name}
          price={p.price}
          category={p.category}
        />
      ))}
    </div>
  );
}
Why .toLowerCase() on both sides?

Case-insensitive match. Without it, typing "book" would miss "Books" because the comparison is exact. Lowercase the haystack, lowercase the needle, compare. Same trick I use in Python with .lower(). There are fancier options (locale-aware collation, unicode normalization) but for an ASCII product list it doesn't matter.

Why compute "visible" before the return instead of inline?

Readability. The {visible.map(...)} in the JSX reads cleanly; {products.filter(...).map(...)} inline gets noisy fast, especially once Exercise 4 adds the category filter as a second .filter() in the chain. Pull the derived value out, name it. Same instinct as breaking up a Rust iterator chain into intermediate let bindings when it gets long.

Where does visible "live"?

It's a regular const inside the function body. It exists for one render, then it's gone. Next render rebuilds it from the current products and current searchTerm. This is the pattern: state holds the inputs, derived values are recomputed on every render from those inputs. No useState for visible, because there's nothing to remember between renders -- it can always be recomputed.

What I noticed running it

Typed "head", got just the headphones. Typed "book", got both books. Cleared the input, got all five back. The list updates on every keystroke and it's instant. Yesterday's "UI = f(state)" thing is now the shape of every interaction: the state is searchTerm, the function is "filter products by it and render," the UI is whatever falls out.

Smaller thing I want to remember: the input has a placeholder, not a label. For a real form I'd want a proper <label> for accessibility. Skipping it in lab is fine; in PEEK's TUI mockups I'd be more careful, because screen-reader users are exactly the kind of "different operator on the same machine" the project is built to think about.

What I'd build on top

Debounce the input. Right now every keystroke filters the list, which is free at five items and expensive at five thousand. setTimeout the actual filter call, cancel on the next keystroke, fire it 200ms after the user stops typing. Don't need it here, but it's the kind of "scales the same approach without changing the model" detail I'd want to know I can add when it's needed.
