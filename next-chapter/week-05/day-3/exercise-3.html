Search + Filter

Exercise 3 / 7 . Level 2 . 30 min

Add a search box. Type, the list narrows. The original products array doesn't change. What changes is which subset of it gets handed to .map() on each render. The list is being projected through a filter, not mutated.

The change

function ProductList() {
  const [searchTerm, setSearchTerm] = useState("");

  const visible = products.filter((p) =>
    p.name.toLowerCase().includes(searchTerm.toLowerCase())
  );

  return (
    <div>
      <h2>Products</h2>

      <input
        type="text"
        value={searchTerm}
        onChange={(e) => setSearchTerm(e.target.value)}
        placeholder="Search products..."
      />

      {visible.map((p) => (
        <ProductCard
          key={p.id}
          name={p.name}
          price={p.price}
          category={p.category}
        />
      ))}
    </div>
  );
}
Why .toLowerCase() on both sides?

Case-insensitive match. Without it, typing "book" would miss "Books" because the comparison is exact. Lowercase the haystack, lowercase the needle, compare. Same trick I use in Python with .lower(). There are fancier options (locale-aware collation, unicode normalization) but for an ASCII product list it doesn't matter.

Why compute "visible" before the return instead of inline?

Readability. The {visible.map(...)} in the JSX reads cleanly. {products.filter(...).map(...)} inline gets noisy fast, especially once Exercise 4 adds the category filter as a second .filter() in the chain. Pull the derived value out, name it. Same instinct as breaking up a Rust iterator chain into intermediate let bindings when it gets long.

Where does visible "live"?

It is a regular const inside the function body. It exists for one render, then it is gone. Next render rebuilds it from the current products and current searchTerm. State holds the inputs, derived values are recomputed on every render from those inputs. No useState for visible, because there is nothing to remember between renders. It can always be recomputed.

Running it

Typed "head", got just the headphones. Typed "book", got both books. Cleared the input, got all five back. Updates on every keystroke and it is instant.

Smaller thing I want to remember. The input has a placeholder, not a label. For a real form I would want a proper <label> for accessibility. Skipping it in lab is fine. In PEEK's TUI mockups I would be more careful, because screen-reader users are exactly the kind of "different operator on the same machine" the project is built to think about.

Twenty more minutes

Debounce the input. Right now every keystroke filters the list, which is free at five items and expensive at five thousand. setTimeout the actual filter call, cancel on the next keystroke, fire it 200ms after the user stops typing. Don't need it here. The kind of detail I want to know I can add when it's needed.
