Controlled Form

Exercise 1 / 7 . Level 1 . 25 min

First controlled-input pass. The "controlled" framing finally made sense when I wrote it instead of reading about it. React owns the value. The input just displays whatever is in state. The onChange handler is the only way the value moves. There is no input.value anywhere. The input is a window into state, not a place where data lives.

The form

import { useState } from "react";

function ContactForm() {
  const [name, setName] = useState("");
  const [email, setEmail] = useState("");
  const [submitted, setSubmitted] = useState(false);

  function handleSubmit(e) {
    e.preventDefault();
    alert(`name: ${name}\nemail: ${email}`);
    setName("");
    setEmail("");
    setSubmitted(true);
  }

  return (
    <div>
      <h2>Contact Form</h2>
      <form onSubmit={handleSubmit}>
        <label>
          Name
          <input
            type="text"
            value={name}
            onChange={(e) => setName(e.target.value)}
          />
        </label>
        <label>
          Email
          <input
            type="email"
            value={email}
            onChange={(e) => setEmail(e.target.value)}
          />
        </label>
        <button type="submit">Submit</button>
      </form>

      <div>
        <h3>Preview:</h3>
        <p>Name: {name || "(empty)"}</p>
        <p>Email: {email || "(empty)"}</p>
        {submitted && <p>Thanks -- form submitted.</p>}
      </div>
    </div>
  );
}
What does e.preventDefault() do here?

Stops the browser's default form-submit behavior, which is to navigate to the form's action URL (defaulting to the current page) and reload. In a SPA you never want that. Everything stays in JS, the page doesn't blink, state survives. Forget the line once and you will watch your whole app disappear into a refresh on first submit. Which is exactly how I learned to remember it.

Why use || "(empty)" in the preview?

Cosmetic. {name} renders nothing when name is the empty string, which collapses the line and makes the layout twitch as the first character lands. The || falls back to a placeholder so the surrounding spacing stays put. Not strictly necessary, just less jumpy.

Running it

Preview updates on every keystroke. React calls the function again on every character, rebuilds the virtual DOM, diffs, patches the <p> text node. At a thousand characters per second that would be a problem. At human typing speed it is invisible. A week ago I would have called that wasteful. Today I get why it is the model.

Footgun I hit on the first pass. I added onChange on the email input and forgot the value prop. The input looked like it worked. Typing went into it fine. But setEmail("") on submit didn't clear it, because the input wasn't actually being driven from state. It was just emitting change events into nowhere. The clear silently no-op'd. That is the uncontrolled-versus-controlled split right there. Both bindings or it is not really controlled.

Twenty more minutes

Basic validation. Disable submit until both fields are non-empty and email has an "@". One line of derived state (const isValid = name && email.includes("@")) and a disabled={!isValid} on the button. Same controlled-input shape, just one more value computed off the existing state. Don't need a form library yet.
