Controlled Form

Exercise 1 / 7 . Level 1 . 25 min

First controlled-input pass. The "controlled" framing finally made sense when I wrote it instead of read it: React owns the value, the input just displays whatever's in state, the onChange handler is the only way the value moves. There is no "input.value" anywhere -- the input is a window into the state, not a place where data lives.

The form

import { useState } from "react";

function ContactForm() {
  const [name, setName] = useState("");
  const [email, setEmail] = useState("");
  const [submitted, setSubmitted] = useState(false);

  function handleSubmit(e) {
    e.preventDefault();
    alert(`name: ${name}\nemail: ${email}`);
    setName("");
    setEmail("");
    setSubmitted(true);
  }

  return (
    <div>
      <h2>Contact Form</h2>
      <form onSubmit={handleSubmit}>
        <label>
          Name
          <input
            type="text"
            value={name}
            onChange={(e) => setName(e.target.value)}
          />
        </label>
        <label>
          Email
          <input
            type="email"
            value={email}
            onChange={(e) => setEmail(e.target.value)}
          />
        </label>
        <button type="submit">Submit</button>
      </form>

      <div>
        <h3>Preview:</h3>
        <p>Name: {name || "(empty)"}</p>
        <p>Email: {email || "(empty)"}</p>
        {submitted && <p>Thanks -- form submitted.</p>}
      </div>
    </div>
  );
}
What does e.preventDefault() do here?

Stops the browser's default form-submit behavior, which is to navigate to the form's action URL (defaulting to the current page) and reload. In a SPA you never want that. Everything stays in JS, the page doesn't blink, state survives. Forget the line once and you'll watch your whole app disappear into a refresh on first submit -- which is exactly how I learned to remember it.

Why use || "(empty)" in the preview?

Cosmetic. {name} renders nothing when name is the empty string, which collapses the line awkwardly. The || falls back to a literal placeholder so the layout doesn't twitch as the user starts typing. Small detail, but the kind of thing that separates a sketch from a real form.

What I noticed running it

The preview updates on every keystroke. That's the controlled-input promise made real. React is calling the function again on every character, rebuilding the virtual DOM, diffing, patching the <p> text node. At thousands of times per second that would matter; at typing speed it's invisible. Yesterday I would have called that wasteful. Today I get why it's the model: simpler reasoning, fast enough by default, expensive only when you build something that's actually expensive.

One footgun I hit: I forgot to add the value prop on the email input the first time, only the onChange. The input still worked for typing, but setEmail("") on submit didn't clear it -- because the input wasn't actually controlled, it was just listening to changes. The clear silently failed. That's the uncontrolled / controlled split right there. You have to bind both directions or React isn't really in control.

What I'd build on top

Basic validation -- disable the submit button until both fields are non-empty and email contains an "@". That's a const isValid = name && email.includes("@") and a disabled={!isValid} on the button. Same controlled-input mental model, just one more derived value computed from state. The pattern scales.
