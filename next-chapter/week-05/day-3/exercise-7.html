First Codex CLI Session

Exercise 7 / 7 . Level 2 . 35 min . Phase 2 gate prep -- session 1

Codex CLI is not new shape for me. I have been running Claude Code as my daily agent since Phase 1, so the "agent in the terminal that reads files and writes diffs" model is the same. What is new is putting it through the cohort's session-discipline rubric explicitly: scope, prompt, review, decide, log. Anyone can install a CLI. The Phase 2 gate is about whether the loop above stays tight.

Setup was the documented one-liner. npm install -g @openai/codex, export OPENAI_API_KEY=... (TA-issued cohort key, not my own), codex inside the to-do project folder. Two-line session header, then the prompt.

The four-step method

1. Read & sketch. Opened src/TodoApp.jsx in vim, read it through. Wrote one sentence in a scratch file: "Persist todos to localStorage so it survives a page refresh; load on mount, save on every change. Only this file." The "only this file" was a deliberate fence. localStorage is small enough to spill into other files if I let the agent run.

2. Try myself first. Before prompting I sketched the shape on paper. useState initializer reads from localStorage with a JSON.parse and a fallback seed, plus a useEffect that writes JSON.stringify(todos) on every change. Two changes, both inside the existing file. Knowing the shape before asking is what turns "did the agent do the right thing?" into a yes/no instead of a vibe check.

3. Use Codex CLI strategically. The prompt I sent: 

In src/TodoApp.jsx, persist `todos` to localStorage so it survives
a page refresh. Use the existing useState + useEffect patterns
already in this file. Initialize from localStorage on mount (fall
back to the current seed array if the key is missing or unparseable),
and write on every change. Only modify TodoApp.jsx. Show me the diff
before applying.

Three things load-bearing in there. File name, "existing patterns" (so it doesn't reach for a library), "only modify this file" (so it doesn't decide to refactor on the way through). Last sentence asks for the diff before applying, which is the gate I want to keep open every session.

4. Reflect. Diff came back mostly right. The useState initializer was a lazy initializer (passing a function, not a value, which is the right shape because the JSON parse shouldn't run on every render). The useEffect had [todos] as the dependency array. Both right. But the agent had also reformatted addTodo. Changed the arrow style and re-indented an unrelated line. Exactly the "touched something I didn't expect" thing the lecture warned about.

Keep / Reject / Modify

Modify. Re-prompted: "Keep only the localStorage changes. Revert the changes to addTodo." Second diff was clean. Just the initializer and the new useEffect. Applied. Refreshed the page, todos survived. Cleared localStorage in DevTools, refreshed, fell back to the seed. Both branches of the fallback worked on the first try.

# CLI Session Log -- Week 5 Day 3 (session 1)

**Goal (one sentence):**
Persist `todos` to localStorage in src/TodoApp.jsx, load on mount, save on every change, no other files.

**Prompt:**
[as above]

**What came back (summary of the diff):**
Lazy useState initializer reading from localStorage with try/catch + seed fallback.
useEffect with [todos] dep writing JSON.stringify on change.
Plus: an unrelated reformat of addTodo (rejected).

**Decision:** Modify
Asked for the localStorage changes only, revert the addTodo touch.

**Result:**
Second diff was clean, applied, verified both branches (seeded localStorage + empty localStorage).

**One thing I'd do differently next session:**
Put "do not modify any other functions in this file, even formatting" in the original
prompt instead of catching the unrelated reformat in review. Cheaper to fence the
agent up front than to reject and re-prompt.

The session-one lesson

The unrelated reformat is the single thing I want to remember from this. The agent is helpful. It sees a thing it thinks could be tidier and tidies it. That is fine in isolation. It is not fine in a session where the whole game is "is this diff exactly what I asked for, no more, no less." The fence has to be in the prompt, because it can't be in the review unless I am willing to reject and re-prompt every time. The reject-and-reprompt loop works but it spends the rate-limit budget on noise.

The other thing: the prompt that worked was longer than the prompts I usually write. I default to terse because terse is faster. But terse means the agent has to guess at the fence, and when it guesses wrong I pay for it on the back end. The cohort's "scope, context, review" rubric is really "spend up front so you spend less on the back end." Same time, different distribution. I would rather think for two minutes before sending than reject for two minutes after.

Logging this honestly is the part I am going to have to discipline. The temptation is to write down the sessions that went well and quietly skip the ones that went sideways. The opposite is the right move. The sideways ones are the ones I will learn from in Week 8.
