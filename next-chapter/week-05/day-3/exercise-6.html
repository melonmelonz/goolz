Component Decomposition

Exercise 6 / 7 . Level 3 . 45 min

Take the monolithic TodoApp from Ex 5 and split it into pieces. The interesting part isn't the cutting -- it's deciding where the seams go and where state lives. The rule I worked from: state lives at the lowest common ancestor of every component that needs to read it or change it. todos belongs in TodoApp because both TodoStats and TodoList need to read it, and TodoForm needs to change it.

The tree

TodoApp                  <- owns todos, defines addTodo / toggle / delete
 |-- TodoForm            <- owns inputValue, calls onAdd(text)
 |-- TodoStats           <- reads remaining count
 \-- TodoList            <- reads todos
      \-- TodoItem       <- reads one todo, calls onToggle(id) / onDelete(id)

The split

function TodoForm({ onAdd }) {
  const [inputValue, setInputValue] = useState("");

  function handleSubmit(e) {
    e.preventDefault();
    if (inputValue.trim() === "") return;
    onAdd(inputValue);
    setInputValue("");
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={inputValue}
        onChange={(e) => setInputValue(e.target.value)}
        placeholder="next bring-up step..."
      />
      <button type="submit">Add</button>
    </form>
  );
}

function TodoStats({ remaining }) {
  return <p>{remaining} item{remaining !== 1 ? "s" : ""} remaining</p>;
}

function TodoItem({ todo, onToggle, onDelete }) {
  return (
    <li>
      <input
        type="checkbox"
        checked={todo.completed}
        onChange={() => onToggle(todo.id)}
      />
      <span style={{
        textDecoration: todo.completed ? "line-through" : "none"
      }}>{todo.text}</span>
      <button onClick={() => onDelete(todo.id)}>Delete</button>
    </li>
  );
}

function TodoList({ todos, onToggle, onDelete }) {
  return (
    <ul>
      {todos.map((t) => (
        <TodoItem
          key={t.id}
          todo={t}
          onToggle={onToggle}
          onDelete={onDelete}
        />
      ))}
    </ul>
  );
}

function TodoApp() {
  const [todos, setTodos] = useState([ /* same seed */ ]);

  function addTodo(text) {
    setTodos([...todos, { id: Date.now(), text, completed: false }]);
  }
  function toggleTodo(id) {
    setTodos(todos.map((t) =>
      t.id === id ? { ...t, completed: !t.completed } : t
    ));
  }
  function deleteTodo(id) {
    setTodos(todos.filter((t) => t.id !== id));
  }

  const remaining = todos.filter((t) => !t.completed).length;

  return (
    <div>
      <h1>BeaglePlay TODO</h1>
      <TodoForm onAdd={addTodo} />
      <TodoList todos={todos} onToggle={toggleTodo} onDelete={deleteTodo} />
      <TodoStats remaining={remaining} />
    </div>
  );
}
Why does TodoForm own its own inputValue, but TodoApp own todos?

Scope of who-needs-it. inputValue is only meaningful inside the form -- nobody else reads it, nobody else changes it. Pulling it up to TodoApp would be lifting state for no reason. todos is read by TodoList and TodoStats and mutated by all three operations from TodoForm, so it belongs at the common parent. Rule of thumb: state goes as low as possible without being inaccessible to the components that need it.

Why pass onAdd as a callback instead of giving TodoForm setTodos directly?

Encapsulation. TodoForm doesn't need to know how a todo gets added -- it just needs to fire "here is the text, the parent figures out the rest." If I later swap setTodos for a reducer, or add server-sync, or validate before insert, the form code doesn't change. The callback boundary is the seam where the form's responsibility ends.

Did splitting it make anything worse?

A little, honestly. The original 50-line monolith was easier to read end-to-end. The split is 90 lines across four functions and the data flow is more abstract -- you have to jump between files (in a real project) to follow a click through to a state update. The tradeoff is worth it once any of these pieces gets reused or once the file would otherwise grow past a screen. At three components and one file the win is the practice, not the production payoff.

What I noticed

The seam between TodoList and TodoItem is the one that feels most "right." TodoList only knows about the collection, TodoItem only knows about one item plus two callbacks. Neither one knows what the state looks like in TodoApp. If I wanted to swap todos for a paginated remote query tomorrow, TodoItem wouldn't notice. That's exactly the decoupling I get out of trait objects in Rust -- the caller and the callee agree on the interface, not the implementation.

The seam I'm least sure about is the inline styling on TodoItem. Pulling the strike-through style up to CSS would let TodoItem stop carrying display logic and just toggle a class. Reasonable refactor for next pass. Not doing it now because the exercise wasn't about CSS.
