Category Filter + Empty State

Exercise 4 / 7 . Level 2 . 30 min

Stack a category filter on top of the search filter. Add a "No products found" state when nothing matches. This is the exercise where conditional rendering and list rendering meet. The UI has to handle "here are results," "here are some results," and "here are no results" without any of those branches feeling tacked on.

The chained filter

const CATEGORIES = ["All", "Electronics", "Books", "Clothing"];

function ProductList() {
  const [searchTerm, setSearchTerm] = useState("");
  const [category, setCategory] = useState("All");

  const visible = products
    .filter((p) =>
      p.name.toLowerCase().includes(searchTerm.toLowerCase())
    )
    .filter((p) =>
      category === "All" ? true : p.category === category
    );

  return (
    <div>
      <h2>Products</h2>

      <input
        type="text"
        value={searchTerm}
        onChange={(e) => setSearchTerm(e.target.value)}
        placeholder="Search products..."
      />

      <div className="cat-row">
        {CATEGORIES.map((c) => (
          <button
            key={c}
            onClick={() => setCategory(c)}
            style={{
              fontWeight: c === category ? "bold" : "normal",
              borderColor: c === category ? "#5E957A" : "#C7C5BF",
            }}
          >
            {c}
          </button>
        ))}
      </div>

      {visible.length === 0 ? (
        <p>No products found.</p>
      ) : (
        visible.map((p) => (
          <ProductCard
            key={p.id}
            name={p.name}
            price={p.price}
            category={p.category}
          />
        ))
      )}
    </div>
  );
}
Why two .filter() calls instead of one combined predicate?

Readability and composition. One filter does one thing (search), the next filter does one thing (category). If I add a third (price range, in stock, etc.), it slots into the chain without rewriting the predicate. Performance is the same to within a rounding error at this size. If the list were millions of rows I would think about a single pass. At five rows the cost of two passes is invisible and the code is cleaner.

Why the ternary for "All" instead of a special case?

The "All" button has to mean "skip the category filter." Returning true for every item is the cleanest way to express that without branching out of the chain. category === "All" ? true : p.category === category reads in English as "if category is All keep it, otherwise keep it when categories match." One line, no early return, the chain stays intact.

Why ternary for the empty state instead of two AND expressions?

Either works. The ternary makes the "exactly one of these renders" relationship explicit, which is what I actually mean. {visible.length === 0 && ...} {visible.length > 0 && ...} would do the same thing but it asks the reader to check that the two conditions are exhaustive and disjoint. The ternary makes that structural by construction.

Running it

Clicked "Books," typed "react." Got "React Handbook." Cleared the search, still on Books, got both. Switched to "Clothing" with nothing typed, got "Running Shoes." Searched "headphones" with "Clothing" still selected, got "No products found." That last one is the empty state actually firing. Firing for the right reason (category disagrees) is what I was testing. The fix in that case is the user picking "All," not the app being smart.

The active-button styling is doing too much inline. For real I would lift those styles into CSS classes, a .cat-btn.active, but inline was faster for the lab.
