Render a Product List

Exercise 2 / 7 . Level 1 . 25 min

Array of objects in, array of JSX elements out, React renders the array. The piece I had to internalize: key isn't a prop the component uses. It is a hint to React's diffing algorithm. The component never reads it. React reads it.

The data + the component

const products = [
  { id: 1, name: "Wireless Headphones", price: 49.99, category: "Electronics" },
  { id: 2, name: "JavaScript Textbook",  price: 29.99, category: "Books" },
  { id: 3, name: "Running Shoes",        price: 89.99, category: "Clothing" },
  { id: 4, name: "USB-C Cable",          price: 12.99, category: "Electronics" },
  { id: 5, name: "React Handbook",       price: 34.99, category: "Books" },
];

function ProductCard({ name, price, category }) {
  return (
    <div className="product-card">
      <h3>{name}</h3>
      <p className="price">${price.toFixed(2)}</p>
      <span className="tag">{category}</span>
    </div>
  );
}

function ProductList() {
  return (
    <div>
      <h2>Products</h2>
      {products.map((p) => (
        <ProductCard
          key={p.id}
          name={p.name}
          price={p.price}
          category={p.category}
        />
      ))}
    </div>
  );
}
Why .map() and not a for loop?

JSX has to evaluate to an expression, not a statement. A for loop is a statement. You can't drop it inside {} and have JSX work with it. .map() is an expression that returns an array, and JSX renders arrays of elements just fine. Same reason you use ternaries instead of if statements inside JSX.

Why key={p.id} and not key={index}?

Index-as-key works for static lists but breaks for dynamic ones. Delete the first item and every remaining item shifts up by one, including its key, because the key was "position." React thinks every item changed, throws away their state, and re-mounts everything. With a stable id, React knows item id=3 is still item id=3 even if its position moved, and shifts the existing DOM nodes instead of rebuilding them. Cheap insurance against a class of bugs you don't see until you're shipping.

Why .toFixed(2) on the price?

Float formatting. 49.99 happens to render as "49.99" but 49.9 would render as "49.9" and 49 as "49". A product list with mixed price formatting looks broken. .toFixed(2) coerces to a string with exactly two decimal places. Real apps use a money library for currency math, never a float, but for display formatting at this level it is fine.

Running it

Five cards rendered, clean. Opened React DevTools and watched it: ProductList at the top, five ProductCard children below, each with its own props row. Each card is its own component instance with its own slot in the tree. That part is invisible from the JSX but obvious once you see the tree. I didn't write five cards. I wrote one component that React instantiated five times.

Tried console.warn-ing inside ProductCard and counted. Five logs on first render, zero on subsequent renders. Nothing changed, so React skipped re-rendering. The "everything re-renders on every state change" worry I had yesterday turns out to be "everything re-renders if its inputs changed, otherwise no."
