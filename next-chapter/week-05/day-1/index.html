Peer Demo

Pair partner asked me to explain npm install. I walked it: you run the command, npm reads package.json for the dep list (or takes the name you passed as an argument), reaches out to the npm registry over HTTPS, downloads tarballs into node_modules/, resolves the transitive tree so every dep's deps are also there, writes the exact resolved versions into package-lock.json so the install is reproducible, and updates the dependencies block in package.json with the version range it just locked. The thing I tried to land hardest: node_modules is generated output. Never edit it. Never commit it. .gitignore it on day one.

Their follow-up: "what happens if I delete node_modules?" Run npm install again and it rebuilds the whole tree from the lockfile. The folder is recoverable; the lockfile is the truth.

Exit Ticket

Q1 -- What is Node.js?

Node is the V8 JavaScript engine -- the same one inside Chrome -- packaged as a standalone runtime that runs on your computer instead of inside a browser tab. The language is exactly the same JavaScript I've been writing for four weeks. What changes is the environment around it: instead of a DOM and a fetch API sandboxed inside a page, Node gives the JS engine access to the filesystem, network sockets, child processes, environment variables -- the actual operating system. That matters because everything past the front of the restaurant lives on a server, and a server is just a program that runs outside a browser. Without Node (or something like it) there's no way for JavaScript to be the server too. Node is what makes "full stack JavaScript" a sentence that means something.

Q2 -- What is npm and what problem does it solve?

npm is the package manager for the JavaScript world. Same job as pip in Python, cargo in Rust, dnf on my Fedora box. It solves two problems at once. First: I don't have to rewrite code other people already wrote and tested. If I need to parse dates, work with timestamps, send HTTP requests, validate emails, there's a package for it -- one command pulls it in. Second, and arguably more important: it solves a coordination problem. package.json records exactly which packages this project depends on, and package-lock.json records the exact versions that got installed. Six months from now on a different machine, npm install reproduces the same tree. Without that, every "works on my machine" bug is just version drift.

Q3 -- What does package.json do? Name at least two things it tracks.

It's the project's manifest -- the file that tells npm (and any other tool) what this project is and what it needs. Two things it tracks that I use every day:

  1. Dependencies. The "dependencies" block lists every package the project needs to run, plus the version range each is allowed to be. "devDependencies" is the same idea for packages only needed during development (test runners, type definitions, build tools).
  2. Scripts. The "scripts" block defines named shortcuts. npm run dev or npm test aren't built-in commands -- they map to whatever shell command you wrote under that key. Half of working in a JS codebase is reading the scripts block to figure out how to actually start the thing.

It also tracks the project name, version, the entry point ("main"), the module system ("type": "module" vs CommonJS), license, author. But deps and scripts are the load-bearing fields.

Q4 -- Predict the output of node app.js.

Trace it:

  • double(7) returns 7 * 2 = 14. console.log prints 14.
  • shout("hello") calls "hello".toUpperCase() which is "HELLO", then concatenates "!!!", giving "HELLO!!!".

Output:

14
HELLO!!!

Two functions exported from utils.js, two functions imported into app.js by name, two calls, two lines printed. The relative path ./utils.js is the bit that always trips people up -- the ./ and the .js are both required when using native ESM in Node.

The Thing That Clicked Today

Modules. Not the syntax -- the syntax is fine. What clicked is that import / export is the same primitive every modern language has, just under different names. Python's from x import y. Rust's use crate::x::y. The fact that JavaScript got it late is a quirk of history, not a sign the idea is hard. Once you have modules, you can stop writing five-hundred-line files and start writing twenty-line files that call each other. That's the unlock.

The TypeScript intro at the end of the day was the right call, too. Twenty minutes of : number and : string and the Codex output I'll see all week stops looking like a foreign language. I'd rather pay that tax now than be confused for a week.
