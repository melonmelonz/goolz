Day 5 -- Shipping: Cloudflare Pages + Serverless

NCD . WK5 . D5 . the thing goes live

All week I have been building components, fetching data, wiring state. None of it is on the internet. It runs on localhost and dies when I close the terminal. Friday is about fixing that. The project is mute-pixel, a pixel art gallery for a game called MUTE. Static HTML, a Preact front end with no build step, Cloudflare Pages Functions for the serverless backend, KV for persistence, and a third-party API (PixelLab) for generating new pixel art on demand. The whole thing deploys with one command. No CI pipeline, no Docker, no YAML.

Cloudflare Pages is the part that surprised me. I expected "static hosting" to mean "dump files on a CDN." It does that, but it also runs serverless functions if you put JavaScript files in a functions/ directory. No extra config. The runtime figures it out. So the static gallery and the API that generates new art live in the same repo, deploy together, and share a KV namespace. I am used to the Rust/kernel world where "it just works" means someone buried the complexity in a proc macro you will need to understand eventually. Here the complexity is genuinely gone. The function runs at the edge, cold-starts in milliseconds, and I never think about servers.

Morning Warm-up -- Where Does Your Code Run?

Three deployment models:

1. Static: HTML/CSS/JS served from a CDN. No server logic. GitHub Pages, Netlify, Cloudflare Pages (without functions). The browser does all the work.

2. Server: A process runs 24/7 (Node, Django, Rails). Handles every request. You pay for idle time. You manage the box or the container.

3. Serverless/Edge: Functions spin up per-request at the CDN edge. No idle cost, no server management. Cloudflare Workers, Pages Functions, AWS Lambda@Edge. The constraint: short-lived, stateless by default, so you lean on KV or a database for persistence.

mute-pixel is model 3. The static gallery is a CDN asset. The /api/generate and /api/gallery endpoints are Pages Functions that run at the edge closest to the user.

Standup

What I did yesterday: finished the React + API exercises. Wired JSONPlaceholder into a component with the loading/data/error pattern. The three-state discipline from D4 is already muscle memory: useState for each state, useEffect with empty deps for the initial fetch, finally to guarantee the loading flag flips.

What I'm doing today: deploying a real project to Cloudflare Pages. Not a tutorial app. A gallery site with a passphrase-gated generator that calls PixelLab's API, stores results in KV, and displays them to anyone who visits. Preact instead of React because the whole thing is a single HTML file with ESM imports. No webpack, no vite, no build.

Where I'm stuck: not stuck, but I am watching one thing. The wrangler pages deploy . command uploads everything in the working directory. If there are large build artifacts (Rust target/ dirs, node_modules) it will try to upload them and choke. Learned that the hard way on another project. The fix is to move them out before deploying or add them to .gitignore. No .deployignore exists for Pages, to my knowledge.

The Stack

Frontend: Single-file Preact app. import { render, h } from 'https://esm.sh/preact@10.23.2' and import htm from 'https://esm.sh/htm@3.1.1' in a <script type="module">. No JSX transform, no bundler. htm tagged templates compile to h() calls at runtime. The trade-off: slightly larger parse cost on first load, but zero build infrastructure to maintain. For a gallery site with 200 assets and a generator form, it is more than enough.

Backend: Two Pages Functions. functions/api/generate.js validates the passphrase, calls PixelLab, stores the base64 PNG in KV. functions/api/gallery.js lists all stored generations. The passphrase is a Cloudflare secret (wrangler pages secret put GATE_SECRET), never in the repo.

Storage: Cloudflare KV. Key pattern: gen:{timestamp}:{uuid-slice}. Value: JSON with prompt, timestamp, cost, and the base64 image. KV is eventually consistent and read-optimized, which is the right fit. Writes are rare (someone generates art), reads are frequent (everyone views the gallery).

Deploy: npx wrangler pages deploy . --project-name mute-pixel. That is the whole pipeline. No git hook, no CI. The Pages project has Git Provider set to "No" so git push does not trigger a deploy. Deliberate. I want to control when the site updates.

The Generator

The community generator is the interesting part. A visitor types a prompt ("a cracked alien obelisk with glowing runes"), picks an asset type (game object, character, environment, tile, freeform), and hits generate. The backend prepends context based on the asset type: "top-down game object sprite, transparent background, pixel art, MUTE game style" gets concatenated with whatever the user typed. The idea is that every generation looks like it belongs in the MUTE world without the user needing to know how to prompt for pixel art.

The passphrase gate is not security. It is credit protection. PixelLab charges per generation. The passphrase is shared with people I trust not to burn through the balance. The constant-time compare in the function is probably overkill for a shared secret that anyone with the passphrase can use, but it costs nothing and the habit is worth keeping.

What I Would Do Differently

KV stores the full base64 PNG inline in the JSON value. For a gallery that grows over time this is going to get expensive on reads. Each gallery load fetches every stored image. The right move is R2 (Cloudflare's object store) for the image blobs and KV for the metadata with an R2 key reference. I skipped it because KV alone is simpler and the gallery has maybe a dozen community generations right now. When it hits 50+ images the gallery endpoint will start feeling slow and that is when R2 earns its setup cost.

I also tried using PixelLab's generate-with-style-v2 endpoint with the original Doug concept art as a style reference image. The results were too constrained; everything came back super pixelated, matching the reference resolution instead of the requested output size. Backed off to text-only style guidance via create-image-pixflux. The text prompt does enough. Sometimes the simpler tool is the right tool.

The Thing That Clicked Today

Deploying is not a separate skill from building. The function that handles POST /api/generate is 80 lines of JavaScript. The KV read in the gallery endpoint is four lines. The Preact front end is one HTML file. The deploy command is one line. There is no "ops" layer here. The code is the infrastructure. I write a function, put it in functions/, and it exists at the edge. The mental model from the kernel work, where deploying means cross-compiling, flashing a board, praying the device tree is right, and watching serial output for panics, made me expect deployment to be a separate phase with its own failure modes. For edge serverless it is not. The failure modes are in the code, not in the deploy. Which means I can iterate on a live site as fast as I can iterate on localhost. That changes how I think about when to ship.
